# E03-S06 — CRUD de procedimento de calibração (versionado)

## Objetivo

Permitir que o responsável técnico gerencie os procedimentos de calibração do laboratório — os documentos técnicos que descrevem como calibrar instrumentos de um determinado domínio metrológico. Cada procedimento é versionado: uma nova versão não sobrescreve a anterior, permitindo que certificados emitidos referenciem a versão exata do procedimento vigente à época.

Nesta story o foco é o CRUD e o controle de versões. A vinculação de procedimento a ordens de serviço e o cálculo de incerteza de medição são implementados em E05.

## Escopo

- Model `ProcedimentoCalibracao` com campos: `tenant_id`, `nome`, `codigo` (identificador único por tenant, ex: "PG-DIM-001"), `versao` (string semântica, ex: "1.0", "1.1", "2.0"), `dominio_metrologico` (enum: `dimensional`, `pressao`, `massa`, `temperatura`), `descricao`, `validade` (date, nullable — prazo de revisão obrigatória), `status` (enum: `rascunho`, `vigente`, `obsoleto`), `publicado_em` (timestamp nullable), `publicado_por_user_id` (FK nullable)
- Par `(codigo, versao)` único por tenant
- Fluxo de versões: publicar uma nova versão (`status = vigente`) marca automaticamente a versão anterior do mesmo `codigo` como `obsoleto`
- Rascunho: procedimento pode ser salvo como `rascunho` sem tornar-se vigente — permite edição antes de publicação
- Listagem paginada mostrando apenas a versão mais recente de cada `codigo` por padrão; parâmetro `?historico=true` traz todas as versões
- Soft-delete lógico via `status = obsoleto` (não remove dados; versões obsoletas ficam acessíveis para referência histórica)
- Policy RBAC: `admin` cria/edita/publica/obsoleta; `tecnico` somente leitura; `atendente` sem acesso
- Isolamento de tenant via `tenant_id`
- Migration, factory e seeder básico (1 procedimento com 2 versões)

## Fora de escopo

## Acceptance Criteria

### AC-001 — Criação de procedimento em rascunho
**Dado** que sou admin do tenant A
**Quando** submeto formulário com nome "Procedimento Dimensional 01", código "PG-DIM-001", versão "1.0", domínio `dimensional`, status `rascunho`
**Então** o procedimento é persistido com `status = rascunho` e `publicado_em = null`

### AC-002 — Publicação de procedimento (rascunho → vigente)
**Dado** que existe procedimento "PG-DIM-001 v1.0" com `status = rascunho`
**Quando** admin publica o procedimento
**Então** `status = vigente`, `publicado_em = now()`, `publicado_por_user_id` do admin são persistidos

### AC-003 — Nova versão obsoleta a versão anterior
**Dado** que "PG-DIM-001 v1.0" está `vigente`
**Quando** admin cria e publica "PG-DIM-001 v1.1"
**Então** "PG-DIM-001 v1.1" fica `vigente`, "PG-DIM-001 v1.0" fica `obsoleto`; ambos permanecem no banco

### AC-004 — Unicidade do par (codigo, versao) por tenant
**Dado** que existe "PG-DIM-001 v1.0" no tenant A
**Quando** tento criar outro procedimento com código "PG-DIM-001" e versão "1.0" no mesmo tenant
**Então** retorna erro de validação de duplicata

### AC-005 — Listagem padrão mostra apenas versão mais recente por código
**Dado** que tenant A tem "PG-DIM-001 v1.0" (obsoleto) e "PG-DIM-001 v1.1" (vigente)
**Quando** faço GET em `/procedimentos` sem parâmetro `historico`
**Então** retorna apenas "PG-DIM-001 v1.1"; "v1.0" não aparece

### AC-006 — Listagem com histórico mostra todas as versões
**Dado** que tenant A tem "PG-DIM-001 v1.0" (obsoleto) e "PG-DIM-001 v1.1" (vigente)
**Quando** faço GET em `/procedimentos?historico=true`
**Então** retornam ambas as versões com seus respectivos status

### AC-007 — Filtro por domínio metrológico
**Dado** que tenant A tem procedimentos de múltiplos domínios
**Quando** filtro por domínio `pressao`
**Então** retornam apenas procedimentos com `dominio_metrologico = pressao`

### AC-008 — Rascunho pode ser editado; vigente não pode ser editado
**Dado** que "PG-DIM-001 v1.0" está `vigente`
**Quando** admin tenta editar o campo `descricao` do procedimento vigente via PUT
**Então** retorna erro de validação "procedimento vigente não pode ser editado; crie uma nova versão"

### AC-009 — Isolamento cross-tenant
**Dado** que sou usuário do tenant A
**Quando** faço GET em `/procedimentos/{id}` onde o ID pertence a procedimento do tenant B
**Então** recebo HTTP 404

### AC-010 — RBAC: atendente não tem acesso a procedimentos
**Dado** que sou usuário com role `atendente`
**Quando** faço GET em `/procedimentos`
**Então** recebo HTTP 403

### AC-011 — RBAC: técnico pode listar e consultar procedimentos
**Dado** que sou usuário com role `tecnico`
**Quando** faço GET em `/procedimentos`
**Então** recebo HTTP 200 com a lista

### AC-012 — RBAC: técnico não pode criar nem publicar procedimento
**Dado** que sou usuário com role `tecnico`
**Quando** faço POST para criação de procedimento
**Então** recebo HTTP 403

### AC-013 — Código segue formato validado
**Dado** que submeto código `pg dim 001` (com espaços)
**Quando** o formulário é submetido
**Então** retorna erro de validação (código deve seguir padrão alfanumérico com hífens, sem espaços)

## Dependências

- Nenhuma story de E03 como pré-requisito (procedimento pertence ao tenant, não a cliente ou instrumento)
- E02 completo (auth, RBAC, tenant isolation)

## Riscos

- Controle de versionamento pode gerar confusão de estado se a transação de "publicar nova versão → obsoleta anterior" não for atômica — mitigação: usar transaction de banco nas duas operações
- Procedimentos obsoletos referenciados em OS futuras (E05) precisam permanecer acessíveis — mitigação: soft-delete lógico via `status`, nunca hard-delete (AC-003 cobre isso)

## Rollback

- `dropTable('procedimentos_calibracao')` no `down()` da migration
- Sem side effects externos

## Evidência necessária para aprovação

- Fluxo rascunho → vigente → nova versão demonstrável via teste de integração
- `php artisan migrate:fresh --seed` sem erro (seeder cria procedimento com 2 versões)
