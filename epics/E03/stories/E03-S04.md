# E03-S04 — CRUD de padrão de referência + cadeia de rastreabilidade

## Objetivo

Permitir que o responsável técnico do laboratório gerencie o acervo de padrões de referência — os instrumentos do próprio laboratório usados para calibrar os instrumentos dos clientes. Cada padrão tem um certificado vigente com data de validade e pertence a uma cadeia de rastreabilidade metrológica que o conecta a uma referência primária (RBC ou padrão internacional), conforme exigido pelo REQ-MET-001.

Após esta story, o laboratório consegue documentar quem calibrou seu padrão, quando vence, e qual é o padrão "pai" na cadeia — com profundidade mínima de 2 níveis visível na interface.

## Escopo

- Model `PadraoReferencia` com campos: `tenant_id`, `modelo`, `fabricante`, `numero_serie`, `certificado_numero`, `certificado_validade` (date), `dominio_metrologico` (enum: `dimensional`, `pressao`, `massa`, `temperatura`), `faixa_medicao_min`, `faixa_medicao_max`, `unidade_faixa`, `resolucao`, `resolucao_unidade`, `padrao_pai_id` (FK self-referencial, nullable), `ativo` (boolean), `observacoes`
- Cadeia de rastreabilidade: auto-relacionamento recursivo via `padrao_pai_id`; profundidade ilimitada no banco, mas UI exibe no mínimo 2 níveis (padrão → pai → avô)
- Consulta `GET /padroes/{id}/cadeia` retorna array da cadeia até raiz (onde `padrao_pai_id IS NULL` representa a referência primária externa)
- `numero_serie` único por tenant
- Listagem paginada (20 por página) com filtro por modelo/número de série/domínio e coluna de status de validade (vigente / próximo do vencimento / vencido)
- Soft-delete via `ativo`
- Policy RBAC: `admin` cria/edita/desativa; `atendente` somente leitura; `tecnico` somente leitura
- Migration, factory e seeder básico (ao menos 1 padrão com 2 níveis de cadeia)

## Fora de escopo

- Alertas de vencimento e bloqueio de uso — E03-S05
- GED de certificados PDF do padrão — E10

- Vinculação a Ordem de Serviço — E05

## Acceptance Criteria

### AC-001 — Criação de padrão de referência sem pai (raiz da cadeia)
**Dado** que sou admin do tenant A
**Quando** cadastro padrão com modelo "Bloco padrão INMETRO", número de série "RBC-001", domínio `dimensional`, validade 2025-12-31 e `padrao_pai_id = null`
**Então** o padrão é persistido com `padrao_pai_id = null` (representa referência primária)

### AC-002 — Criação de padrão com pai (nível 2 da cadeia)
**Dado** que existe o padrão "RBC-001" (nível 1) no tenant A
**Quando** cadastro padrão "LAB-001" com `padrao_pai_id = RBC-001`
**Então** o padrão é persistido com `padrao_pai_id = RBC-001` e a cadeia tem 2 níveis

### AC-003 — Criação de padrão com avô (nível 3 da cadeia)
**Dado** que existe a cadeia RBC-001 → LAB-001
**Quando** cadastro "LAB-002" com `padrao_pai_id = LAB-001`
**Então** a cadeia RBC-001 → LAB-001 → LAB-002 existe com 3 níveis

### AC-004 — Consulta da cadeia de rastreabilidade
**Dado** que existe a cadeia RBC-001 → LAB-001 → LAB-002
**Quando** faço GET em `/padroes/LAB-002/cadeia`
**Então** retorno é array `[LAB-002, LAB-001, RBC-001]` (do mais recente ao mais ancestral), cada item com `id`, `modelo`, `numero_serie`, `certificado_validade`

### AC-005 — Prevenção de ciclo na cadeia (auto-referência direta)
**Dado** que o padrão "LAB-001" existe com `padrao_pai_id = RBC-001`
**Quando** tento atualizar "RBC-001" para ter `padrao_pai_id = LAB-001` (criaria ciclo)
**Então** retorna erro de validação "ciclo detectado na cadeia de rastreabilidade" e o update não é persistido

### AC-006 — Unicidade de número de série dentro do tenant
**Dado** que existe padrão com número de série "RBC-001" no tenant A
**Quando** tento cadastrar outro padrão com o mesmo número de série no tenant A
**Então** retorna erro de validação de duplicata

### AC-007 — Isolamento cross-tenant
**Dado** que sou usuário do tenant A
**Quando** faço GET em `/padroes/{id}` onde o ID pertence a padrão do tenant B
**Então** recebo HTTP 404

### AC-008 — Listagem com status de validade calculado
**Dado** que tenant A tem 3 padrões: vencido (validade < hoje), próximo (validade entre hoje e hoje+30 dias), vigente (validade > hoje+30 dias)
**Quando** listo padrões
**Então** cada item tem campo `status_validade` com valor `vencido`, `proximo_vencimento` ou `vigente` respectivamente

### AC-009 — Filtro por domínio metrológico
**Dado** que tenant A tem padrões de múltiplos domínios
**Quando** filtro listagem por domínio `massa`
**Então** retornam apenas padrões com `dominio_metrologico = massa`

### AC-010 — RBAC: atendente não pode criar padrão
**Dado** que sou usuário com role `atendente`
**Quando** faço POST para criação de padrão
**Então** recebo HTTP 403

### AC-011 — RBAC: admin pode criar padrão
**Dado** que sou usuário com role `admin`
**Quando** submeto formulário de criação de padrão com dados válidos
**Então** o padrão é persistido e recebo resposta de sucesso

### AC-012 — RBAC: técnico pode listar e consultar padrões
**Dado** que sou usuário com role `tecnico`
**Quando** faço GET na listagem de padrões do meu tenant
**Então** recebo HTTP 200 com a lista

### AC-013 — Desativação de padrão
**Dado** que existe padrão ativo "LAB-001"
**Quando** admin desativa o padrão
**Então** `ativo = false` é persistido e o padrão não aparece na listagem padrão

## Dependências

- Nenhuma story de E03 como pré-requisito (padrões pertencem ao tenant, não a clientes)
- E02 completo (auth, RBAC, tenant isolation)

## Riscos

- Auto-relacionamento recursivo pode gerar N+1 na consulta da cadeia — mitigação: usar CTE recursivo no PostgreSQL ou carga eager com limite de profundidade (max 10 níveis)
- Ciclo na cadeia de rastreabilidade poderia causar loop infinito — mitigação: validar ausência de ciclo antes de persistir (AC-005 cobre isso)

## Rollback

- `dropTable('padroes_referencia')` no `down()` da migration
- Sem side effects externos

## Evidência necessária para aprovação

- Consulta de cadeia com 3 níveis retornando array correto
- `php artisan migrate:fresh --seed` sem erro (seeder cria cadeia de 2 níveis)
