# E03-S07 — Audit log de cadastros (owen-it/laravel-auditing)

## Objetivo

Registrar automaticamente todas as alterações em dados dos cadastros do E03 — clientes, contatos, instrumentos, padrões de referência e procedimentos de calibração — usando a biblioteca `owen-it/laravel-auditing`. O log captura quem alterou, quando, qual campo mudou, o valor anterior e o valor novo.

Este audit log é exigência do epico (ver `epic.md`) e é pré-condição para compliance com ISO 17025 (rastreabilidade de dados) e LGPD (responsabilização pelo tratamento de dados pessoais). Após esta story, qualquer alteração em um cadastro é auditável por um administrador.

## Escopo

- Instalação e configuração de `owen-it/laravel-auditing` (via Composer; deve estar em `composer.json`)
- Trait `Auditable` adicionada aos models: `Cliente`, `Contato`, `PadraoReferencia`, `ProcedimentoCalibracao`, `Instrumento`
- Campos excluídos do audit por padrão: `updated_at`, `remember_token` (não gera ruído desnecessário)
- Campos de dados pessoais auditados com flag de sensibilidade: `email`, `whatsapp` de `Contato` são incluídos no log, mas o acesso ao log desses campos é restrito a `admin` (não a `tecnico`)
- Endpoint `GET /auditoria/clientes/{id}` — lista todos os eventos de audit do cliente com paginação (20 por página), ordenados por `created_at DESC`
- Endpoint `GET /auditoria/padroes/{id}` — idem para padrões
- Endpoint `GET /auditoria/procedimentos/{id}` — idem para procedimentos
- Endpoint `GET /auditoria/instrumentos/{id}` — idem para instrumentos
- Resposta de cada entrada de audit inclui: `evento` (created/updated/deleted), `user_id`, `user_nome`, `campos_alterados` (array com `campo`, `valor_anterior`, `valor_novo`), `criado_em`
- Isolamento: audit entries são filtradas pelo `tenant_id` do model auditado; usuário só acessa audit de entidades do seu tenant
- Policy RBAC: somente `admin` acessa endpoints de auditoria
- A tabela `audits` é criada pela migration do pacote; o agente não cria migration própria

## Fora de escopo

- Audit de Ordem de Serviço e calibrações — E05
- Audit de usuários e permissões — E02 (já coberto pelo RBAC do E02)
- Exportação de audit log em CSV — pós-MVP
- Log imutável em storage externo — pós-MVP (a tabela `audits` é append-only por convenção, mas não tem garantia criptográfica nesta story)
- Purga automática de audit antigo — pós-MVP (RNF-009 define retenção de 2 anos para logs de acesso)

## Acceptance Criteria

### AC-001 — Audit registra criação de cliente
**Dado** que o audit log está ativo
**Quando** atendente cria um novo cliente
**Então** um registro de audit é criado com `event = created`, `auditable_type = Cliente`, `auditable_id = {id_do_cliente}`, `user_id = {id_do_atendente}`

### AC-002 — Audit registra atualização de cliente com valores anterior e novo
**Dado** que o cliente `C001` tem `razao_social = "Empresa Antiga Ltda"`
**Quando** atendente atualiza `razao_social` para "Empresa Nova Ltda"
**Então** um registro de audit é criado com `event = updated`, contendo `old_values.razao_social = "Empresa Antiga Ltda"` e `new_values.razao_social = "Empresa Nova Ltda"`

### AC-003 — Audit registra desativação de cliente
**Dado** que cliente `C001` está ativo
**Quando** atendente desativa o cliente (altera `ativo = false`)
**Então** um registro de audit é criado com `event = updated`, `old_values.ativo = true`, `new_values.ativo = false`

### AC-004 — Audit registra criação e atualização de contato
**Dado** que o audit log está ativo
**Quando** atendente cria um contato e depois edita o campo `papel`
**Então** existem 2 registros de audit para o contato: um `created` e um `updated` com os valores do campo `papel`

### AC-005 — Audit registra criação e atualização de padrão de referência
**Dado** que o audit log está ativo
**Quando** admin cria um padrão e depois edita `certificado_validade`
**Então** existem 2 registros de audit para o padrão com os respectivos eventos

### AC-006 — Audit registra publicação de procedimento
**Dado** que "PG-DIM-001 v1.0" está em rascunho
**Quando** admin publica o procedimento (altera `status` de `rascunho` para `vigente`)
**Então** um registro de audit é criado com `old_values.status = rascunho` e `new_values.status = vigente`

### AC-007 — Endpoint de auditoria de cliente retorna histórico paginado
**Dado** que o cliente `C001` tem 25 registros de audit
**Quando** admin faz GET em `/auditoria/clientes/C001`
**Então** retornam 20 registros ordenados por `created_at DESC` com metadados de paginação

### AC-008 — Isolamento: admin só vê audit de entidades do seu tenant
**Dado** que sou admin do tenant A
**Quando** faço GET em `/auditoria/clientes/{id}` onde o cliente pertence ao tenant B
**Então** recebo HTTP 404

### AC-009 — RBAC: somente admin acessa endpoints de auditoria
**Dado** que sou usuário com role `atendente` ou `tecnico`
**Quando** faço GET em qualquer endpoint `/auditoria/*`
**Então** recebo HTTP 403

### AC-010 — Campos `updated_at` não geram entradas de audit isoladas
**Dado** que uma atualização ocorre em um modelo auditado
**Quando** consulto o audit log
**Então** o campo `updated_at` NÃO aparece em `old_values` ou `new_values` (excluído da auditoria)

### AC-011 — Audit de instrumento registra criação e edição
**Dado** que o audit log está ativo nos models `Instrumento`
**Quando** atendente cria um instrumento e depois edita `numero_serie`
**Então** existem 2 registros de audit para o instrumento: `created` e `updated` com campo `numero_serie`

### AC-012 — Usuário removido não quebra o audit log
**Dado** que existe um registro de audit cujo `user_id` pertence a um usuário que foi removido do tenant
**Quando** admin acessa o audit log
**Então** o registro é exibido com `user_id` preservado e `user_nome = "(usuário removido)"` (ou similar) — sem erro 500

## Riscos

- `owen-it/laravel-auditing` pode ter incompatibilidade com versão do Laravel em uso — mitigação: verificar compatibilidade na tabela de versões do pacote antes de instalar; fixar versão no `composer.json`
- Volume de audit entries pode crescer rapidamente — mitigação: índice em `(auditable_type, auditable_id)` na tabela `audits` (normalmente criado pela migration do pacote); monitorar tamanho em RNF-009

## Rollback

- `composer remove owen-it/laravel-auditing`
- `artisan migrate:rollback` desfaz a migration da tabela `audits` (criada pelo pacote)
- Remoção do trait `Auditable` dos models via `git revert`
