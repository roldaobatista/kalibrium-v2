# E03-S01 — CRUD de cliente com validação CNPJ/CPF

## Objetivo

Permitir que o atendente do laboratório cadastre, consulte, edite e desative clientes (pessoa jurídica via CNPJ ou pessoa física via CPF) dentro do seu tenant. O cliente é a entidade raiz que ancora contatos, instrumentos e ordens de serviço — sem ele nenhum fluxo operacional pode começar.

Após esta story, o atendente consegue localizar um cliente existente em uma lista paginada, abrir seu cadastro completo (razão social, endereço, regime tributário, limite de crédito) e ter a certeza de que o CNPJ ou CPF informado é válido antes de salvar.

## Escopo

- Model `Cliente` com campos: `cnpj_cpf`, `tipo_pessoa` (PJ/PF), `razao_social`, `nome_fantasia`, `endereço completo` (logradouro, número, complemento, bairro, cidade, UF, CEP), `regime_tributario` (Simples, Lucro Presumido, Lucro Real, MEI, Isento), `limite_credito` (decimal), `ativo` (boolean)
- Validação algorítmica de CNPJ (dígitos verificadores) — sem chamada externa à Receita Federal no MVP
- Validação algorítmica de CPF (dígitos verificadores)
- CNPJ/CPF único por tenant (não pode haver duplicata dentro do mesmo tenant)
- Listagem paginada de clientes (20 por página) com filtro por razão social/CNPJ/CPF e ordenação por `razao_social`
- Soft-delete via campo `ativo` (cliente desativado não aparece na listagem padrão, mas dados são preservados)
- Policy RBAC: somente roles `admin` e `atendente` podem criar/editar/desativar; `tecnico` tem somente leitura
- Isolamento de tenant: cliente pertence ao tenant via `tenant_id`; queries filtradas por scope global
- Migration, factory e seeder básico

## Fora de escopo

- Contatos do cliente — E03-S02
- Importação em massa via CSV — pós-MVP
- Consulta à API da Receita Federal — pós-MVP (fallback algorítmico é suficiente para MVP)
- Histórico de alterações (audit log) — E03-S07
- Vinculação a instrumentos — E03-S03
- GED de documentos do cliente — E10

## Acceptance Criteria

### AC-001 — Criação de cliente PJ com CNPJ válido
**Dado** que sou um usuário com role `atendente` autenticado no tenant A
**Quando** submeto o formulário com CNPJ `11.222.333/0001-81` (válido algoritmicamente), razão social e endereço completo
**Então** o cliente é persistido no banco com `tipo_pessoa = PJ`, `tenant_id` do tenant A, e a resposta HTTP retorna 201 ou redirect com flash de sucesso

### AC-002 — Criação de cliente PF com CPF válido
**Dado** que sou um usuário com role `atendente`
**Quando** submeto formulário com CPF `529.982.247-25` (válido algoritmicamente) e nome completo
**Então** o cliente é persistido com `tipo_pessoa = PF` e os dados ficam acessíveis via rota de exibição

### AC-003 — Rejeição de CNPJ inválido
**Dado** que sou um usuário com role `atendente`
**Quando** submeto formulário com CNPJ `11.111.111/1111-11` (inválido)
**Então** a request retorna erro de validação no campo `cnpj_cpf` e nenhum registro é persistido

### AC-004 — Rejeição de CPF inválido
**Dado** que sou um usuário com role `atendente`
**Quando** submeto formulário com CPF `111.111.111-11` (inválido)
**Então** a request retorna erro de validação no campo `cnpj_cpf` e nenhum registro é persistido

### AC-005 — Unicidade de CNPJ/CPF dentro do tenant
**Dado** que já existe um cliente com CNPJ `11.222.333/0001-81` no tenant A
**Quando** tento cadastrar um segundo cliente com o mesmo CNPJ no tenant A
**Então** a request retorna erro de validação indicando duplicata

### AC-006 — Isolamento de CNPJ entre tenants
**Dado** que existe um cliente com CNPJ `11.222.333/0001-81` no tenant A
**Quando** um usuário do tenant B cadastra um cliente com o mesmo CNPJ
**Então** o cadastro é aceito (CNPJ único apenas dentro do tenant, não globalmente)

### AC-007 — Listagem paginada com isolamento de tenant
**Dado** que tenant A tem 25 clientes e tenant B tem 10 clientes
**Quando** um usuário do tenant A lista clientes (página 1, 20 por página)
**Então** retorna exatamente 20 registros, todos com `tenant_id` do tenant A, sem nenhum registro do tenant B

### AC-008 — Filtro por razão social
**Dado** que tenant A tem clientes com razões sociais variadas
**Quando** filtro por substring da razão social (ex: "Calibra")
**Então** retorna apenas clientes cujo `razao_social` ou `nome_fantasia` contém a substring (case-insensitive)

### AC-009 — Edição de cliente
**Dado** que existe um cliente no tenant A
**Quando** o atendente edita o `limite_credito` e o `regime_tributario`
**Então** os novos valores são persistidos e a exibição reflete a alteração

### AC-010 — Desativação de cliente (soft-delete)
**Dado** que existe um cliente ativo no tenant A
**Quando** o atendente desativa o cliente
**Então** `ativo = false` é persistido, o cliente não aparece na listagem padrão (`WHERE ativo = true`), mas continua acessível via rota de exibição direta com ID

### AC-011 — RBAC: técnico não pode criar cliente
**Dado** que sou um usuário com role `tecnico`
**Quando** faço POST para a rota de criação de cliente
**Então** recebo HTTP 403 e nenhum registro é persistido

### AC-012 — RBAC: técnico pode listar e ver cliente
**Dado** que sou um usuário com role `tecnico`
**Quando** faço GET na listagem e em um cliente específico do meu tenant
**Então** recebo HTTP 200 com os dados do cliente

### AC-013 — Cross-tenant: cliente de outro tenant retorna 404
**Dado** que sou um usuário do tenant A
**Quando** faço GET na rota `/clientes/{id}` onde o ID pertence a um cliente do tenant B
**Então** recebo HTTP 404 (o scope global filtra e o registro não é encontrado)

## Dependências

- E02 completo (auth, RBAC, multi-tenancy com scope global de tenant)

## Riscos

- Algoritmo de validação CNPJ/CPF deve cobrir casos-limite (CNPJ com zeros, CPF com dígitos repetidos) — mitigação: usar biblioteca `laravellegends/laravel-validation-rules` ou implementar regra customizada com suite de casos de teste
- Soft-delete pode conflitar com constraint de unicidade de CNPJ — mitigação: incluir `ativo` no índice único ou usar `whereNull`/`where ativo=true` na rule de unique

## Rollback

- Migration tem `down()` com `dropTable('clientes')` — reversível via `artisan migrate:rollback`
- Sem side effects externos (nenhuma chamada à Receita Federal no MVP)

- Listagem renderiza com paginação visível em ambiente local
