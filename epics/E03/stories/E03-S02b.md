# E03-S02b — Consentimentos LGPD + imutabilidade + validações

## Objetivo

Registrar os consentimentos LGPD para cada canal de comunicação (e-mail marketing, WhatsApp) por contato, com imutabilidade de log e rastreabilidade completa. Após esta story o laboratório tem base legal documentada antes de enviar qualquer comunicação automática, atendendo ao Art. 7º e Art. 8º da LGPD.

## Escopo

- Model `ConsentimentoLGPD` com campos: `contato_id`, `canal` (enum: `email_marketing`, `whatsapp`), `concedido` (boolean), `concedido_em` (timestamp), `revogado_em` (timestamp nullable), `forma_aceite` (texto livre), `registrado_por_user_id`
- Um contato pode ter N consentimentos (um por canal); apenas o mais recente por canal é considerado ativo
- Opt-out: atendente pode revogar consentimento — `revogado_em` é preenchido, `concedido = false`; registro original não é deletado (imutabilidade de log)
- Novo consentimento após revogação cria novo registro sem apagar os anteriores
- Migration, factory e seeder básico

## Fora de escopo

- CRUD de contato (criação, listagem, RBAC) — E03-S02a
- Envio efetivo de e-mail ou WhatsApp — E06 (notificações)
- Exportação de dados para portabilidade LGPD (Art. 18) — pós-MVP
- Exclusão de dados por direito ao esquecimento — pós-MVP

## Criterios de aceite

### AC-003 — Registro de consentimento de e-mail marketing
**Dado** que existe o contato `CT001` no tenant A
**Quando** o atendente marca consentimento para canal `email_marketing` com forma de aceite "formulário web"
**Então** um registro `ConsentimentoLGPD` é criado com `canal = email_marketing`, `concedido = true`, `concedido_em = now()`, `registrado_por_user_id` do atendente atual

### AC-004 — Registro de consentimento de WhatsApp
**Dado** que existe o contato `CT001` com WhatsApp preenchido
**Quando** o atendente marca consentimento para canal `whatsapp`
**Então** um registro `ConsentimentoLGPD` é criado com `canal = whatsapp`, `concedido = true`

### AC-005 — Revogação de consentimento (opt-out)
**Dado** que o contato `CT001` tem consentimento ativo para `email_marketing`
**Quando** o atendente revoga o consentimento
**Então** o registro original tem `revogado_em = now()` e `concedido = false`; o registro anterior NÃO é deletado

### AC-006 — Imutabilidade do log de consentimento
**Dado** que existe um registro de `ConsentimentoLGPD` revogado
**Quando** consulto o histórico de consentimentos do contato
**Então** vejo tanto o registro original (com `concedido_em`) quanto o registro de revogação (com `revogado_em`); nenhum dos dois foi deletado

### AC-007 — Consentimento duplicado cria novo registro sem apagar o anterior
**Dado** que o contato `CT001` revogou consentimento de `email_marketing`
**Quando** o atendente registra novo consentimento para `email_marketing`
**Então** um novo registro `ConsentimentoLGPD` é criado (`concedido = true`); os registros anteriores permanecem no banco; o mais recente é considerado o estado ativo

## Riscos

- Forma de aceite como texto livre pode dificultar auditoria — mitigação: definir valores pré-definidos recomendados na UI, mas armazenar como string para flexibilidade
- Enum de canal deve ser extensível para futuros canais (SMS, push) — mitigação: string com constraint de check, não tinyint fixo
