# E03-S02 — CRUD de contato + consentimentos LGPD

## Objetivo

Permitir que o atendente vincule um ou mais contatos a um cliente cadastrado, registrando o papel de cada contato (comprador, responsável técnico) e os consentimentos LGPD para cada canal de comunicação ativo (e-mail marketing, WhatsApp). Após esta story, o laboratório tem base legal documentada antes de enviar qualquer comunicação automática a um contato.

O registro de consentimento é explícito: data, canal, forma de aceite e quem registrou ficam gravados de forma auditável — atendendo ao Art. 7º e Art. 8º da LGPD e à política interna `docs/security/lgpd-base-legal.md`.

## Escopo

- Model `Contato` com campos: `cliente_id`, `nome`, `email`, `whatsapp` (E.164 ou formato brasileiro), `papel` (enum: `comprador`, `responsavel_tecnico`, `outro`), `ativo` (boolean)
- Model `ConsentimentoLGPD` com campos: `contato_id`, `canal` (enum: `email_marketing`, `whatsapp`), `concedido` (boolean), `concedido_em` (timestamp), `revogado_em` (timestamp nullable), `forma_aceite` (texto livre: "formulário web", "verbal registrado pelo atendente", etc.), `registrado_por_user_id`
- Um contato pode ter N consentimentos (um por canal); apenas o mais recente por canal é considerado ativo
- Opt-out: atendente pode revogar consentimento — `revogado_em` é preenchido, `concedido = false`; registro original não é deletado (imutabilidade de log)
- Listagem de contatos por cliente (sem paginação se ≤ 10; paginada se > 10)
- Um cliente deve ter ao menos 1 contato ativo (validação de negócio — warn, não block)
- Policy RBAC: `admin` e `atendente` criam/editam/revogam; `tecnico` somente leitura
- Isolamento via `cliente_id` → `tenant_id` (contato herda tenant do cliente)
- Migration, factory e seeder básico

## Fora de escopo

- Envio efetivo de e-mail ou WhatsApp — E06 (notificações)
- Exportação de dados para portabilidade LGPD (Art. 18) — pós-MVP
- Exclusão de dados por direito ao esquecimento — pós-MVP (requer fluxo de solicitação formal)
- Audit log de alterações de contato — E03-S07

## Acceptance Criteria

### AC-001 — Criação de contato vinculado a cliente
**Dado** que existe o cliente `C001` no tenant A e sou atendente
**Quando** submeto formulário com nome, e-mail e papel `comprador`
**Então** o contato é persistido com `cliente_id = C001` e `ativo = true`

### AC-002 — Múltiplos contatos por cliente
**Dado** que o cliente `C001` já tem 1 contato
**Quando** cadastro um segundo contato com papel `responsavel_tecnico`
**Então** ambos os contatos existem vinculados ao cliente e aparecem na listagem

### AC-003 — Registro de consentimento de e-mail marketing
**Dado** que existe o contato `CT001` no tenant A
**Quando** o atendente marca consentimento para canal `email_marketing` com forma de aceite "formulário web"
**Então** um registro `ConsentimentoLGPD` é criado com `canal = email_marketing`, `concedido = true`, `concedido_em = now()`, `registrado_por_user_id` do atendente atual

### AC-004 — Registro de consentimento de WhatsApp
**Dado** que existe o contato `CT001` com WhatsApp preenchido
**Quando** o atendente marca consentimento para canal `whatsapp`
**Então** um registro `ConsentimentoLGPD` é criado com `canal = whatsapp`, `concedido = true`

### AC-005 — Revogação de consentimento (opt-out)
**Dado** que o contato `CT001` tem consentimento ativo para `email_marketing`
**Quando** o atendente revoga o consentimento
**Então** o registro original tem `revogado_em = now()` e `concedido = false`; o registro anterior NÃO é deletado; nenhum novo envio de e-mail marketing deve ser possível para esse contato

### AC-006 — Imutabilidade do log de consentimento
**Dado** que existe um registro de `ConsentimentoLGPD` revogado
**Quando** consulto o histórico de consentimentos do contato
**Então** vejo tanto o registro original (com `concedido_em`) quanto o registro de revogação (com `revogado_em`); nenhum dos dois foi deletado

### AC-007 — Consentimento duplicado cria novo registro sem apagar o anterior
**Dado** que o contato `CT001` revogou consentimento de `email_marketing`
**Quando** o atendente registra novo consentimento para `email_marketing`
**Então** um novo registro `ConsentimentoLGPD` é criado (`concedido = true`); os registros anteriores permanecem no banco; o mais recente é considerado o estado ativo

### AC-008 — Isolamento: contato de outro tenant inacessível
**Dado** que sou usuário do tenant A
**Quando** faço GET em `/contatos/{id}` onde o ID pertence a contato de cliente do tenant B
**Então** recebo HTTP 404

### AC-009 — RBAC: técnico não pode criar nem editar contato
**Dado** que sou usuário com role `tecnico`
**Quando** faço POST para criação de contato
**Então** recebo HTTP 403

### AC-010 — RBAC: técnico pode listar contatos do seu tenant
**Dado** que sou usuário com role `tecnico`
**Quando** faço GET em `/clientes/{id}/contatos` para um cliente do meu tenant
**Então** recebo HTTP 200 com a lista de contatos

### AC-011 — Desativação de contato
**Dado** que existe o contato `CT001` ativo
**Quando** o atendente desativa o contato
**Então** `ativo = false` é persistido e o contato não aparece na listagem padrão de contatos ativos

### AC-012 — Validação de formato de WhatsApp
**Dado** que submeto whatsapp `99999-9999` (sem DDD)
**Quando** o formulário é submetido
**Então** retorna erro de validação no campo `whatsapp` (deve ter DDD + número, mínimo 10 dígitos numéricos)

## Dependências

- E03-S01 (modelo `Cliente` deve existir para `contato.cliente_id`)

## Riscos

- Definição de "canal" de consentimento pode precisar de extensão futura (SMS, push) — mitigação: enum extensível (não usar tinyint fixo; usar string com constraint de check)
- Forma de aceite como texto livre pode dificultar auditoria — mitigação: definir valores pré-definidos recomendados na UI, mas armazenar como string para flexibilidade

## Rollback

- `dropTable('consentimentos_lgpd')` seguido de `dropTable('contatos')` no `down()` da migration
- Sem side effects externos
