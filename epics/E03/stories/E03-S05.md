# E03-S05 — Alertas de vencimento de padrão + bloqueio de uso

## Objetivo

Implementar os mecanismos automáticos que detectam padrões de referência próximos do vencimento ou já vencidos, notificam o responsável técnico via alerta na interface e bloqueiam o uso de padrões vencidos em novas ordens de serviço. Isso garante rastreabilidade metrológica contínua: nenhuma calibração pode ser emitida com base em um padrão sem certificado válido.

O alerta é disparado quando `certificado_validade < hoje + 30 dias`. O bloqueio é ativado quando `certificado_validade < hoje` — o padrão não pode ser selecionado em nenhuma OS.

## Escopo

- Job/Command `VerificarVencimentoPadroes` agendado via Laravel Scheduler para rodar diariamente às 06:00 (horário de Brasília): varre todos os padrões ativos de todos os tenants, dispara evento `PadraoReferenciaProximoVencimento` (30+ dias antes) e `PadraoReferenciaVencido` (no dia ou após vencimento)
- Listener `NotificarAdminVencimentoPadrao`: escuta os dois eventos e cria registro na tabela `alertas_sistema` com `tipo`, `padrao_referencia_id`, `tenant_id`, `visto_em` (nullable)
- Alerta visível na interface: badge de contagem no menu principal para admin/tecnico; listagem de alertas pendentes (`visto_em IS NULL`)
- Marcar alerta como visto: ação de usuário que preenche `visto_em = now()` e `visto_por_user_id`
- Bloqueio de uso: method `PadraoReferencia::estaVigente()` retorna `false` quando `certificado_validade < hoje`; toda seleção de padrão em OS (E05) deve chamar este método — nesta story, o bloqueio é implementado no model; a integração com OS é feita em E05
- Endpoint `GET /padroes/vencidos` lista padrões com `certificado_validade < hoje` e `ativo = true` (para relatório administrativo)
- Endpoint `GET /padroes/proximos-vencimento` lista padrões com `certificado_validade BETWEEN hoje AND hoje+30_dias` e `ativo = true`
- Isolamento de tenant: job roda por tenant; alertas são criados com `tenant_id` correto; usuário só vê alertas do seu tenant
- Policy RBAC: alertas visíveis para `admin` e `tecnico`; `atendente` não vê alertas de padrão

## Fora de escopo

- Envio de e-mail de alerta — E06 (notificações por canal externo)
- Renovação automática de certificado — fluxo manual, sem automação MVP
- Integração com OS para bloqueio efetivo — E05 (o method `estaVigente()` existe, mas E05 o usa)
- Audit log dos alertas — E03-S07

## Acceptance Criteria

### AC-001 — Job detecta padrão próximo do vencimento (janela 30 dias)
**Dado** que existe padrão "LAB-001" com `certificado_validade = hoje + 29 dias` no tenant A
**Quando** o job `VerificarVencimentoPadroes` é executado manualmente via `artisan app:verificar-vencimento-padroes`
**Então** um registro é criado em `alertas_sistema` com `tipo = proximo_vencimento`, `padrao_referencia_id = LAB-001`, `tenant_id = tenant_A`, `visto_em = null`

### AC-002 — Job detecta padrão vencido
**Dado** que existe padrão "LAB-002" com `certificado_validade = hoje - 1 dia` no tenant A
**Quando** o job é executado
**Então** um registro é criado em `alertas_sistema` com `tipo = vencido`, `padrao_referencia_id = LAB-002`

### AC-003 — Padrão ainda dentro da validade não gera alerta
**Dado** que existe padrão "LAB-003" com `certificado_validade = hoje + 60 dias`
**Quando** o job é executado
**Então** nenhum alerta é criado para LAB-003

### AC-004 — Job idempotente: não cria alerta duplicado
**Dado** que já existe alerta `vencido` para "LAB-002" de uma execução anterior do job hoje
**Quando** o job executa novamente no mesmo dia
**Então** nenhum alerta duplicado é criado para LAB-002 (o job verifica se já existe alerta `visto_em IS NULL` do mesmo tipo para o mesmo padrão)

### AC-005 — Alerta visível na interface para admin
**Dado** que existem 2 alertas não vistos no tenant A
**Quando** o admin do tenant A acessa qualquer página da aplicação
**Então** um badge com a contagem "2" é exibido no menu principal

### AC-006 — Alerta não visível para atendente
**Dado** que existem alertas de padrão vencido no tenant A
**Quando** um usuário com role `atendente` acessa a aplicação
**Então** o badge de alerta de padrão NÃO aparece no menu (atendente não gerencia padrões)

### AC-007 — Marcar alerta como visto
**Dado** que existe alerta não visto `A001` no tenant A
**Quando** admin faz PATCH em `/alertas/A001/visto`
**Então** `visto_em = now()` e `visto_por_user_id` do admin são persistidos; o alerta não aparece mais na contagem de pendentes

### AC-008 — Isolamento de alertas entre tenants
**Dado** que sou admin do tenant A
**Quando** faço GET em `/alertas`
**Então** retornam apenas alertas com `tenant_id = tenant_A`; alertas do tenant B não aparecem

### AC-009 — Method `estaVigente()` retorna false para padrão vencido
**Dado** que existe padrão "LAB-002" com `certificado_validade = hoje - 1 dia`
**Quando** invoco `$padrao->estaVigente()`
**Então** o método retorna `false`

### AC-010 — Method `estaVigente()` retorna true para padrão dentro da validade
**Dado** que existe padrão "LAB-001" com `certificado_validade = hoje + 29 dias`
**Quando** invoco `$padrao->estaVigente()`
**Então** o método retorna `true` (próximo do vencimento mas ainda vigente)

### AC-011 — Endpoint de padrões vencidos retorna apenas vencidos do tenant
**Dado** que sou admin do tenant A, com 2 padrões vencidos e 3 vigentes
**Quando** faço GET em `/padroes/vencidos`
**Então** retornam exatamente 2 registros, todos do tenant A

### AC-012 — Scheduler configurado para 06:00 BRT
**Dado** que o Scheduler do Laravel está ativo
**Quando** verifico `app/Console/Kernel.php` ou `routes/console.php`
**Então** o job `VerificarVencimentoPadroes` está agendado com `->dailyAt('06:00')->timezone('America/Sao_Paulo')`

## Dependências

- E03-S04 (model `PadraoReferencia` com campo `certificado_validade` deve existir)

## Riscos

- Job rodando em contexto multi-tenant pode vazar dados entre tenants se o escopo global não estiver ativo — mitigação: job usa `Tenant::all()` e inicializa contexto de tenant antes de cada query; teste AC-008 valida o isolamento
- Idempotência do job pode ser complexa se o mesmo padrão mudar de estado entre execuções — mitigação: lógica simples: "existe alerta não-visto deste tipo para este padrão? então não cria"

## Rollback

- `dropTable('alertas_sistema')` no `down()` da migration
- Remover agendamento do Scheduler via `git revert`
- Sem side effects externos (nenhum e-mail ou webhook nesta story)

## Evidência necessária para aprovação

- Testes Pest cobrindo todos os ACs passando com exit 0
- `php artisan app:verificar-vencimento-padroes` roda sem erro com output de padrões processados
- Badge de alerta visível na interface quando há padrões próximos do vencimento
