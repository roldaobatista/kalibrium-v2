# E03-S06a — Model procedimento de calibração + fluxo de versões

## Objetivo

Criar o model `ProcedimentoCalibracao` com controle de versões: publicar uma nova versão torna a anterior obsoleta atomicamente. Após esta story o laboratório consegue gerenciar procedimentos técnicos com rastreabilidade de versão, condição para compliance com ISO 17025.

## Escopo

- Model `ProcedimentoCalibracao` com campos: `tenant_id`, `nome`, `codigo` (identificador único por tenant), `versao` (string semântica, ex: "1.0", "1.1"), `dominio_metrologico` (enum: `dimensional`, `pressao`, `massa`, `temperatura`), `descricao`, `validade` (date, nullable), `status` (enum: `rascunho`, `vigente`, `obsoleto`), `publicado_em` (timestamp nullable), `publicado_por_user_id` (FK nullable)
- Par `(codigo, versao)` único por tenant
- Validação de `codigo`: formato alfanumérico com hífens, sem espaços
- Fluxo de versões: publicar nova versão (`status = vigente`) marca atomicamente (transaction) a versão anterior do mesmo `codigo` como `obsoleto`
- Rascunho pode ser editado; procedimento `vigente` não pode ser editado (retorna erro de validação)
- Migration, factory e seeder básico (1 procedimento com 2 versões)

## Criterios de aceite

### AC-001 — Criação de procedimento em rascunho
**Dado** que sou admin do tenant A
**Quando** submeto formulário com nome "Procedimento Dimensional 01", código "PG-DIM-001", versão "1.0", domínio `dimensional`, status `rascunho`
**Então** o procedimento é persistido com `status = rascunho` e `publicado_em = null`

### AC-002 — Publicação de procedimento (rascunho → vigente)
**Dado** que existe procedimento "PG-DIM-001 v1.0" com `status = rascunho`
**Quando** admin publica o procedimento
**Então** `status = vigente`, `publicado_em = now()`, `publicado_por_user_id` do admin são persistidos

### AC-003 — Nova versão obsoleta a versão anterior
**Dado** que "PG-DIM-001 v1.0" está `vigente`
**Quando** admin cria e publica "PG-DIM-001 v1.1"
**Então** "PG-DIM-001 v1.1" fica `vigente`, "PG-DIM-001 v1.0" fica `obsoleto`; ambos permanecem no banco

### AC-004 — Unicidade do par (codigo, versao) por tenant
**Dado** que existe "PG-DIM-001 v1.0" no tenant A
**Quando** tento criar outro procedimento com código "PG-DIM-001" e versão "1.0" no mesmo tenant
**Então** retorna erro de validação de duplicata

### AC-008 — Rascunho pode ser editado; vigente não pode ser editado
**Dado** que "PG-DIM-001 v1.0" está `vigente`
**Quando** admin tenta editar o campo `descricao` do procedimento vigente via PUT
**Então** retorna erro de validação "procedimento vigente não pode ser editado; crie uma nova versão"

## Testes obrigatórios

- Feature: criação em rascunho persiste com status=rascunho e publicado_em=null
- Feature: publicação preenche status=vigente, publicado_em, publicado_por_user_id

- Feature: tentativa de editar procedimento vigente retorna erro de validação
- Feature: par (codigo, versao) duplicado retorna erro de validação

## Riscos

- Transição "publicar nova versão → obsoleta anterior" deve ser atômica — mitigação: usar transaction de banco nas duas operações
- Versões obsoletas referenciadas em OS futuras precisam permanecer acessíveis — mitigação: soft-delete lógico via `status`, nunca hard-delete
