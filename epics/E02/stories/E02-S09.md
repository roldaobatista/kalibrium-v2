# E02-S09 ‚Äî Gap-fill: Automacao da transicao trial ‚Üí expired/suspended

## Observacao

Story gap-fill criada em 2026-04-17 para fechar **GAP-S05-001** identificado em E02-S05 (documentado em `epics/E02/stories/E02-S05.md ¬ßGaps identificados`).

O modelo `Tenant` tem o campo `trial_ends_at`, mas nao existe job/command que monitore expiracao do trial e realize a transicao automatica `trial ‚Üí suspended` (ou `trial ‚Üí cancelled` conforme politica). Sem este mecanismo, tenants podem permanecer indefinidamente em status `trial` apos o prazo de 30 dias ‚Äî d√≠vida tecnica conhecida para o MVP fechado.

## Objetivo

Implementar automacao diaria da transicao de estado de tenants cuja data `trial_ends_at` ja passou, respeitando a maquina de estados existente (`E02-S05`) e registrando auditoria de cada transicao para fins de dunning/reativacao.

## Escopo

- `app/Console/Commands/ExpireTrialTenants.php` ‚Äî Artisan command idempotente que:
  - Consulta `tenants.where('status', 'trial').where('trial_ends_at', '<=', now())`
  - Aplica transicao para `suspended` (read-only mode conforme E02-S05 AC-002/AC-006)
  - Registra em `tenant_status_transitions` (audit log existente em E02-S05)
  - Envia notificacao ao gerente (email conforme E02-S07 politica LGPD)
  - Saida idempotente: rodar 2x seguidas nao duplica transicoes
- `app/Providers/AppServiceProvider.php` ‚Äî schedule do command via Laravel Scheduler:
  - Frequencia: diaria as 03:00 UTC
  - Timezone: UTC (padrao do sistema)
  - Lock para evitar overlap: `withoutOverlapping(60)`
- Testes unitarios (Pest):
  - `tests/Unit/Commands/ExpireTrialTenantsTest.php` ‚Äî 5 cenarios (trial no prazo, trial expirado, tenant ja suspenso, tenant cancelado, idempotencia)
- Testes de integracao:
  - `tests/Feature/TrialExpirationScheduleTest.php` ‚Äî schedule registrado corretamente
- Documentacao operacional em `docs/operations/trial-expiration-job.md`:
  - Como forcar execucao manual (`php artisan tenants:expire-trial`)
  - Como consultar transicoes recentes via query direta
  - Plano de recuperacao em caso de job falhar por 1+ dia

## Fora de escopo

- Notificacao push (ampliacao E12/E21 ‚Äî depende de APNs/FCM)
- Reativacao automatica pos-pagamento (requer E21 compliance fiscal + gateway)
- UI para forcar expiracao de tenant especifico (console interno pos-MVP)
- Cobranca/gateway de pagamento

## Acceptance Criteria

### AC-001 ‚Äî Command identifica trials expirados e transita para suspended
**Dado** que existem tenants `trial` com `trial_ends_at` no passado, **quando** `php artisan tenants:expire-trial` e executado, **entao** cada um desses tenants transita para status `suspended` e uma linha e adicionada em `tenant_status_transitions` com `from_status=trial, to_status=suspended, reason=trial_expired, triggered_by=system`.
- **Evidencia:** `tests/Unit/Commands/ExpireTrialTenantsTest.php::test_trial_expirado_transita_para_suspended` exit 0

### AC-002 ‚Äî Command nao toca tenants com trial ainda vigente
**Dado** que existe tenant `trial` com `trial_ends_at` no futuro, **quando** o command e executado, **entao** o status permanece `trial` e nenhuma linha e adicionada em `tenant_status_transitions`.
- **Evidencia:** `tests/Unit/Commands/ExpireTrialTenantsTest.php::test_trial_no_prazo_nao_transita` exit 0

### AC-003 ‚Äî Command e idempotente
**Dado** que um tenant ja foi transitado para `suspended` pelo command em execucao anterior, **quando** o command e executado novamente, **entao** nenhuma linha nova e adicionada em `tenant_status_transitions` e o status permanece `suspended`.
- **Evidencia:** `tests/Unit/Commands/ExpireTrialTenantsTest.php::test_idempotencia_duas_execucoes` exit 0

### AC-004 ‚Äî Command nao toca tenants em outros estados
**Dado** que existem tenants com status `active`, `suspended`, `cancelled`, `dunning` ou `reactivated`, **quando** o command e executado, **entao** nenhum deles transita de estado.
- **Evidencia:** `tests/Unit/Commands/ExpireTrialTenantsTest.php::test_outros_estados_ignorados` exit 0

### AC-005 ‚Äî Schedule registrado no Laravel Scheduler
**Dado** que `AppServiceProvider` registra o command no Scheduler, **quando** se executa `php artisan schedule:list`, **entao** a saida contem `tenants:expire-trial` com cron `0 3 * * *`.
- **Evidencia:** `tests/Feature/TrialExpirationScheduleTest.php::test_schedule_registrado` exit 0

### AC-006 ‚Äî Notificacao por email e enviada ao gerente do tenant
**Dado** que um tenant transitou para `suspended` via command, **quando** o evento `TenantTrialExpired` e disparado, **entao** um email `TrialExpiredNotification` e enfileirado para cada usuario com papel `gerente` do tenant ‚Äî verificado via `Mail::fake()`.
- **Evidencia:** `tests/Unit/Commands/ExpireTrialTenantsTest.php::test_email_enviado_ao_gerente` exit 0

### AC-007 ‚Äî Execucao com lock previne overlap
**Dado** que uma instancia do command ja esta rodando, **quando** uma segunda instancia tenta iniciar, **entao** a segunda instancia nao executa (Laravel Scheduler `withoutOverlapping` ativo) ‚Äî verificado pela presenca da diretiva no schedule.
- **Evidencia:** `tests/Feature/TrialExpirationScheduleTest.php::test_withoutOverlapping_configurado`

## Riscos

- **Fuso horario do `trial_ends_at`** ‚Äî mitigacao: usar comparacao em UTC consistente com o que foi persistido em E02-S02
- **Email cair em spam** ‚Äî aceito para MVP; E21 tratara SPF/DKIM formal
- **Timer nao disparar por janela de manutencao** ‚Äî mitigacao: catch-up na proxima execucao (query e declarativa, nao reativa a evento de expiracao exato)

## Estimativa

## Rollback

`git revert` do PR. Remove o command e o schedule. Tenants que ja foram transitados via command permanecem em `suspended` (transicoes ja persistidas em `tenant_status_transitions` e enforced por middleware E02-S05). Para reverter uma transicao especifica, usar migration de correcao manual pontual.
