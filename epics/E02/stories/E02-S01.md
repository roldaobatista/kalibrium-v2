# E02-S01 — Scaffold stancl/tenancy + RLS PostgreSQL

## Observacao
Contrato formal escrito retroativamente em 2026-04-16 para fechar ER-001 da epic-retrospective E02. Implementacao ja mergeada via slice 008 (TEN-001). ACs abaixo mapeados contra tests/codigo existentes; gaps flagados explicitamente.

## Objetivo
Instalar e configurar `stancl/tenancy` em modo single-database com Row-Level Security do PostgreSQL ativo, garantindo que todo acesso a dados sensíveis ao tenant seja automaticamente filtrado por `tenant_id` via global scope Eloquent e RLS nativo do banco.

## Escopo
- Instalacao do pacote `stancl/tenancy` em modo single-database
- Configuracao do PostgreSQL RLS na camada de banco (policies por tabela)
- Global scope Eloquent injetado automaticamente em todos os models sensíveis ao tenant
- Migration base de `tenants` com `tenant_id` como chave de isolamento
- Índice em `tenant_id` em todas as tabelas sensíveis (performance desde o inicio)

## Fora de escopo
- Entidades de negocio (Empresa, Filial) — E02-S02
- Autenticacao e sessao — E02-S03
- RBAC — E02-S04
- Multi-database tenancy (fora do MVP conforme ADR-0001)

## Acceptance Criteria

### AC-001 — stancl/tenancy instalado e configurado em modo single-database
**Dado** o ambiente de aplicacao configurado, **quando** a aplicacao iniciar, **entao** o pacote `stancl/tenancy` deve estar presente em `composer.json` com modo single-database e o provider registrado no bootstrap.
- **Evidencia:** `tests/slice-008/TenantSettingsIsolationTest.php` — testes que pressupõem tenancy ativo; setup de tenant via `TestHelpers`
- **Implementacao:** `config/tenancy.php`, `composer.json`
- **Status:** ✅ covered

### AC-002 — PostgreSQL RLS ativo para tabelas sensíveis
**Dado** dois tenants distintos com dados cadastrados, **quando** qualquer query Eloquent for executada no contexto do tenant A, **entao** apenas dados com `tenant_id = A` devem ser retornados, sem necessidade de filtro explicito na query.
- **Evidencia:** `tests/slice-008/TenantSettingsIsolationTest.php` — `AC-005: dois tenants não se veem` + `tests/slice-011/TenantIsolationModelTest.php`
- **Implementacao:** migrations com `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`, `SetCurrentTenantContext` middleware
- **Status:** ✅ covered

### AC-003 — Global scope Eloquent injetado automaticamente
**Dado** qualquer model sensível ao tenant, **quando** uma query for executada sem filtro explicito de `tenant_id`, **entao** o global scope deve aplicar `WHERE tenant_id = ?` automaticamente.
- **Evidencia:** `tests/slice-011/TenantIsolationModelTest.php` — testa que models sem filtro manual respeitam isolamento
- **Implementacao:** `app/Models/Concerns/BelongsToTenant.php` ou equivalente, `SetCurrentTenantContext` middleware
- **Status:** ✅ covered

### AC-004 — Indice em tenant_id em todas as tabelas sensíveis
**Dado** as migrations executadas, **quando** inspecionar o schema do banco, **entao** todas as tabelas com `tenant_id` devem ter indice no campo.
- **Evidencia:** `tests/slice-011/TenantIsolationPerformanceTest.php`
- **Implementacao:** migrations do slice 008
- **Status:** ✅ covered

### AC-005 — Contexto de tenant propagado para jobs e filas
**Dado** um job disparado dentro de um contexto de tenant, **quando** o job executar, **entao** o `tenant_id` deve ser propagado corretamente sem vazar dados de outro tenant.
- **Evidencia:** `tests/slice-011/TenantIsolationJobTest.php`
- **Implementacao:** `SetCurrentTenantContext` middleware + configuracao do tenancy para jobs
- **Status:** ✅ covered

## Dependencias
- E01 completo (CI, deploy staging)
- ADR-0001 (PostgreSQL, Laravel 13)

## Riscos
- RLS pode causar degradacao de performance em queries complexas — mitigado por índice desde o inicio (AC-004)
- Jobs sem contexto de tenant propagado podem vazar dados — mitigado por AC-005
