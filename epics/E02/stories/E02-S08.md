# E02-S08 — Testes estruturais de isolamento entre tenants

## Objetivo

Criar uma suite de testes de seguranca estrutural que prove — de forma sistematica, nao so por amostra — que dados de um tenant nunca aparecem para outro tenant em qualquer query, endpoint HTTP, job assincrono ou relatorio. Esta suite e a rede de protecao que ampara toda a expansao para E03 e alem.

## Escopo

- **Teste de isolamento por model:** para cada model sensivel ao tenant, criar 2 tenants com dados homonimos e confirmar que o escopo global filtra corretamente em `all()`, `find()`, `where()`, `whereHas()`, `with()`, agregacoes (count/sum/avg) e relacionamentos.
- **Teste de isolamento HTTP:** para cada rota autenticada, autenticar usuario do tenant A e tentar acessar recurso com ID do tenant B. Deve retornar 404 ou 403, nunca 200 com dados cruzados.
- **Teste de isolamento de jobs:** disparar job no contexto do tenant A e garantir que ele nao vaza tenant context quando processa (propagacao correta via middleware stancl/tenancy).
- **Teste de isolamento de relatorios/exports:** PDFs, CSVs e JSONs gerados contem apenas dados do tenant atual.
- **Smoke de regressao:** suite roda em CI em todo PR que toca `app/Models/**`, `app/Http/**`, `app/Livewire/**`, `app/Jobs/**`.
- **Cobertura minima:** 100% dos models com `tenant_id` direto ou indireto.

## Fora de escopo

- Penetration test externo (entrega de governanca, fora do MVP).
- Testes de escalacao de privilegio dentro do mesmo tenant — ja coberto pelas policies RBAC do E02-S04.
- Teste de vazamento via logs/telemetria (follow-up post-MVP).

## Acceptance Criteria

- **AC-001** — Teste `TenantIsolationTest` cobre 100% dos models sensiveis listados em `config/tenancy.php` com 2 tenants + 1 admin de cada.
- **AC-002** — Para cada rota em `routes/web.php` e `routes/api.php` com middleware `auth`, existe teste de cross-tenant que tenta acesso e assert 404/403.
- **AC-003** — Jobs que dependem de contexto de tenant (listados em `config/tenancy-jobs.php`) tem teste que roda no tenant A e valida persistencia no tenant A apenas.
- **AC-004** — Relatorios (PlansPage summary, eventual /reports/*) validados com 2 tenants; export contem apenas dados do tenant autenticado.
- **AC-005** — CI job `tenant-isolation` obrigatorio em PRs que tocam models/controllers/livewire/jobs — falha bloqueia merge.
- **AC-006** — Teste de performance: suite isolamento roda em <60s em CI padrao (aumento linear com N models e N rotas, sem explosao combinatorial).
- **AC-007** — README em `tests/tenant-isolation/README.md` explica como adicionar novos models à suite em 1 parrafo + 1 exemplo.

## Riscos

- Suite pode ficar lenta se cada teste instanciar 2 tenants completos via factories; mitigacao: fixture compartilhada com 2 tenants base + reset incremental por teste.
- Cobertura automatica de "todas as rotas" exige introspeccao das rotas registradas via Artisan — test pode precisar rodar fora do container.
