# E02-S03 — Autenticacao: login, 2FA, recuperacao de senha

## Observacao
Contrato formal escrito retroativamente em 2026-04-16 para fechar ER-001 da epic-retrospective E02. Implementacao ja mergeada via slice 007 (SEG-001). ACs abaixo mapeados contra tests/codigo existentes; gaps flagados explicitamente.

## Objetivo
Implementar o fluxo completo de acesso seguro ao Kalibrium: login por e-mail e senha, 2FA por TOTP obrigatório para papéis gerente e administrativo (por papel, não apenas por flag manual), recuperacao de senha por e-mail com token com prazo, e auditoria de todos os eventos de autenticacao sem persistir segredos.

## Escopo
- Login por e-mail + senha via Laravel Fortify (`POST /auth/login`)
- Redirecionamento para 2FA challenge quando papel exige (`gerente`, `administrativo`)
- 2FA por TOTP com código de recuperacao como fallback (`POST /auth/two-factor-challenge`)
- Recuperacao de senha por e-mail com mensagem neutra (`POST /auth/forgot-password`)
- Reset de senha com token com prazo e senha mínima de 12 caracteres (`POST /auth/reset-password`)
- Rate limiting em tentativas de login (429 apos excesso)
- Auditoria em `login_audit_logs` de todos os eventos (sucesso, falha, bloqueio)
- Bloqueio de acesso para tenant `cancelled`, tenant `suspended` em modo somente leitura
- Bloqueio de usuario com vinculo `suspended`, `invited` ou `removed`

## Fora de escopo
- Registro público de novo laboratorio
- SSO, SAML, SCIM, OIDC Enterprise
- Gestao de usuarios e papeis — E02-S04
- Configuracao de 2FA pelo usuario final (este slice assume 2FA ja configurado no setup)

## Acceptance Criteria

### AC-001 — Login com credenciais válidas cria sessao e redireciona para /app
**Dado** um usuário ativo com tenant `active` ou `trial` e senha correta, **quando** `POST /auth/login` receber e-mail, senha e CSRF válidos, **entao** o sistema autentica, cria sessao web e redireciona para `/app`.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php:~16` — `test('AC-001: login bem-sucedido...')`
- **Implementacao:** `routes/auth.php`, `app/Actions/Fortify/AuthenticateUser.php`
- **Status:** ✅ covered

### AC-002 — Gerente e administrativo sao redirecionados para 2FA challenge apos login
**Dado** um usuário ativo com papel `gerente` ou `administrativo`, **quando** `POST /auth/login` receber credenciais válidas, **entao** o sistema nao abre `/app` e redireciona para `/auth/two-factor-challenge` — inclusive quando o flag `requires_2fa` estiver `false` no vinculo.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php:70` — `test('AC-002: gerente e administrativo exigem 2FA mesmo sem flag manual no vinculo')` com `->with(['gerente', 'administrativo'])`
- **Implementacao:** `app/Actions/Fortify/AuthenticateUser.php` — checagem por papel alem do flag; `app/Http/Middleware/EnsureTwoFactorChallengeCompleted.php`
- **Status:** ✅ covered

### AC-003 — 2FA com TOTP válido conclui autenticacao
**Dado** um usuário com desafio 2FA pendente e código TOTP válido, **quando** `POST /auth/two-factor-challenge` for enviado, **entao** o sistema conclui autenticacao, cria sessao web e redireciona para `/app`.
- **Evidencia:** `tests/slice-007/AuthTwoFactorTest.php:~12` — testa challenge com gerente e TOTP válido
- **Implementacao:** Fortify built-in two-factor challenge handler
- **Status:** ✅ covered

### AC-004 — Codigo de recuperacao 2FA válido conclui autenticacao e invalida o codigo
**Dado** um usuário com desafio 2FA pendente e código de recuperacao válido, **quando** `POST /auth/two-factor-challenge` for enviado com `recovery_code`, **entao** o sistema conclui autenticacao, invalida o código e registra auditoria do uso.
- **Evidencia:** `tests/slice-007/AuthTwoFactorTest.php` — testa recovery code
- **Implementacao:** Fortify built-in recovery code handler
- **Status:** ✅ covered

### AC-005 — Recuperacao de senha retorna mensagem neutra independente de o e-mail existir
**Dado** qualquer e-mail em formato válido, **quando** `POST /auth/forgot-password` for enviado, **entao** o sistema responde com mensagem neutra sem revelar se o e-mail existe.
- **Evidencia:** `tests/slice-007/AuthPasswordResetTest.php` — testa mensagem neutra
- **Implementacao:** Fortify built-in password reset handler com customizacao de mensagem
- **Status:** ✅ covered

### AC-006 — Reset de senha com token válido e senha minima de 12 caracteres
**Dado** um token de recuperacao válido, **quando** `POST /auth/reset-password` receber nova senha com pelo menos 12 caracteres, **entao** o sistema altera a senha, invalida o token e redireciona para `/auth/login`.
- **Evidencia:** `tests/slice-007/AuthPasswordResetTest.php` — testa reset com token válido e senha curta
- **Implementacao:** `app/Actions/Fortify/ResetUserPassword.php` — validacao de 12 caracteres
- **Status:** ✅ covered

### AC-007 — Todos os eventos de autenticacao gravados em login_audit_logs sem segredos
**Dado** qualquer tentativa de login (sucesso ou falha), **quando** o fluxo terminar, **entao** o sistema grava `login_audit_logs` com evento, `user_id`, `tenant_id`, IP e hash de user agent, sem persistir senha, token, segredo TOTP ou código de recuperacao.
- **Evidencia:** `tests/slice-007/AuthAuditTest.php` — testa auditoria de login e ausencia de segredos
- **Implementacao:** `app/Listeners/RecordLoginAudit.php`, tabela `login_audit_logs`
- **Status:** ✅ covered

### AC-008 — Credenciais erradas retornam mensagem neutra
**Dado** e-mail ou senha incorretos, **quando** `POST /auth/login` for enviado, **entao** o sistema retorna erro de validacao com mensagem neutra sem revelar se o e-mail existe.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php` — edge case de credenciais inválidas
- **Implementacao:** `app/Actions/Fortify/AuthenticateUser.php`
- **Status:** ✅ covered

### AC-009 — Rate limit em excesso de tentativas retorna 429
**Dado** excesso de tentativas de login para o mesmo e-mail/IP dentro da janela configurada, **quando** `POST /auth/login` for enviado novamente, **entao** o sistema retorna `429` sem tentar autenticar a senha.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php` — testa rate limit
- **Implementacao:** `config/fortify.php` — throttle de login; Laravel rate limiter
- **Status:** ✅ covered

### AC-010 — Tenant cancelled bloqueia acesso mesmo com credenciais válidas
**Dado** usuário ativo com tenant `cancelled`, **quando** `POST /auth/login` receber credenciais válidas, **entao** o sistema bloqueia acesso, nao cria sessao de aplicacao e registra auditoria do bloqueio.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php` — testa tenant cancelled
- **Implementacao:** `app/Actions/Fortify/AuthenticateUser.php` — checagem de status do tenant
- **Status:** ✅ covered

### AC-011 — Tenant suspended permite autenticacao em modo somente leitura
**Dado** usuário ativo com tenant `suspended`, **quando** `POST /auth/login` receber credenciais válidas, **entao** o sistema autentica, cria sessao em modo somente leitura e registra auditoria do acesso restrito.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php:AC-011`
- **Implementacao:** `app/Http/Middleware/EnsureReadOnlyTenantMode.php`
- **Status:** ✅ covered

### AC-012 — Vinculo nao ativo bloqueia usuário especifico
**Dado** usuário com vinculo `suspended`, `invited` ou `removed`, **quando** `POST /auth/login` receber credenciais válidas, **entao** o sistema bloqueia somente esse usuário, nao cria sessao e registra auditoria neutra.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php:AC-013`
- **Implementacao:** `app/Actions/Fortify/AuthenticateUser.php` — checagem de status do vinculo
- **Status:** ✅ covered

### AC-013 — Rota protegida sem autenticacao redireciona para login sem expor dados
**Dado** usuário nao autenticado, **quando** acessar rota protegida como `/app`, **entao** o sistema redireciona para `/auth/login` sem expor dados de tenant.
- **Evidencia:** `tests/slice-007/AuthLoginTest.php:AC-019`
- **Implementacao:** middleware `auth` no grupo de rotas `/app`
- **Status:** ✅ covered

### AC-014 — Segredos ausentes de qualquer response e log de autenticacao
**Dado** qualquer response dos fluxos de login, recuperacao, reset ou 2FA, **quando** o conteúdo e logs forem inspecionados, **entao** nao aparecem senha, token de reset, segredo TOTP, código TOTP nem código de recuperacao.
- **Evidencia:** `tests/slice-007/AuthAuditTest.php` — AC-021 do spec verificado
- **Implementacao:** `app/Listeners/RecordLoginAudit.php` — nao persiste payload cru
- **Status:** ✅ covered

## Riscos
- 2FA pode bloquear gerente legítimo sem fluxo de recovery code — mitigado por AC-004 e AC-012 (spec 007)
- Mensagens de erro podem enumerar usuarios — mitigado por AC-005 e AC-008
- Auditoria pode vazar credenciais — mitigado por AC-007 e AC-014
