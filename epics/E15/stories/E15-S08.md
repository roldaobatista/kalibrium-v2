# E15-S08 — Device binding: registro de dispositivo + endpoint `/api/v1/devices`

## Objetivo

Implementar o vínculo de identidade entre o usuário e o dispositivo físico que ele usa para acessar o Kalibrium. Cada dispositivo gera um par de chaves ed25519, registra a chave pública no backend via `POST /api/v1/devices`, e a partir daí toda sessão é assinada com a chave privada do dispositivo — permitindo que o admin revogue acesso de um dispositivo específico (wipe remoto, E15-S10) sem invalidar outros dispositivos do mesmo usuário.

Após esta story, o backend tem a tabela `registered_devices` com `tenant_id` (ADR-0016), o app registra o dispositivo no primeiro login, e o admin (Persona 1 — Marcelo) consegue listar os dispositivos registrados de um usuário via tela admin em `/admin/devices`.

## Escopo

### Backend (ampliação de E02 — Laravel)
- Migration nova: tabela `registered_devices` com campos: `id` (uuid), `tenant_id` (obrigatório, indexed), `user_id` (FK users), `device_name` (string, ex: "iPhone 15 de João"), `public_key` (text, ed25519), `platform` (enum: ios/android/web), `last_seen_at` (timestamp), `wiped_at` (nullable timestamp), `created_at`, `updated_at`
- Eloquent Global Scope por `tenant_id` + RLS no PostgreSQL (ADR-0016 — tabela `registered_devices` está na lista das 10 críticas)
- `POST /api/v1/devices` — registra novo dispositivo; autenticação Bearer JWT; body: `{ device_name, public_key, platform }`; resposta: 201 com `{ id, tenant_id, device_name, platform, created_at }`
- `GET /api/v1/devices` — lista dispositivos do usuário logado (scope por `user_id` + `tenant_id`)
- `GET /api/v1/devices` (admin) — lista todos os dispositivos do tenant (somente role `admin`); parâmetro `?user_id=` para filtrar por usuário
- Política RBAC: usuário vê seus próprios dispositivos; admin vê todos do tenant; super_admin vê todos

### Frontend (React + Capacitor)
- Geração do par de chaves ed25519 via `@noble/ed25519` na primeira abertura do app após login
- Armazenamento da chave privada em SecureStorage (Keychain iOS / Keystore Android) — nunca persistida fora do dispositivo
- Chamada automática a `POST /api/v1/devices` no fluxo de primeiro login (após JWT estabelecido em E15-S07)
- Tela `/admin/devices`: listagem de dispositivos por usuário (somente admin), com nome do dispositivo, plataforma, último acesso e botão "Revogar" (chama E15-S10 — stub nesta story)
- Testes: geração de par de chaves, chamada ao endpoint, RBAC na tela admin

## Fora de escopo

- Endpoint de wipe (`POST /api/v1/devices/{id}/wipe`) — E15-S10
- Assinatura de cada request com chave privada (PKI completo) — pós-MVP; nesta story o device binding é para identidade e wipe, não para autenticação de cada request
- Notificação ao usuário quando dispositivo novo é registrado — E12/E21

## Critérios de aceite

### AC-001 — Migration cria tabela `registered_devices` com `tenant_id`
**Dado** que a migration é executada no backend
**Quando** se executa `php artisan migrate` no backend Laravel
**Então** a tabela `registered_devices` existe, tem coluna `tenant_id NOT NULL` e a migration `.down()` remove a tabela sem erros — verificado via Pest: `$this->artisan('migrate:fresh --seed')` com exit 0

### AC-002 — `POST /api/v1/devices` retorna 201 com dispositivo criado
**Dado** que o usuário tem JWT válido
**Quando** o app faz `POST /api/v1/devices` com `{ device_name: "iPhone 15 de João", public_key: "<ed25519 pubkey>", platform: "ios" }`
**Então** o backend retorna HTTP 201 com body contendo `id`, `tenant_id`, `device_name`, `platform` e `created_at` — verificado via Pest no backend e `tests/unit/device/register.spec.ts` no frontend com mock

### AC-003 — Dispositivo registrado pertence ao `tenant_id` correto
**Dado** que o usuário do tenant A faz `POST /api/v1/devices`
**Quando** o dispositivo é criado
**Então** o registro em `registered_devices` tem `tenant_id` do tenant A — não de outro tenant — verificado via Pest

### AC-004 — Isolamento: usuário do tenant B não vê dispositivos do tenant A
**Dado** que tenant A tem 2 dispositivos registrados e tenant B tem 1
**Quando** um usuário do tenant B chama `GET /api/v1/devices`
**Então** a resposta contém somente o dispositivo do tenant B (total: 1) — verificado via Pest em `tests/Feature/Devices/DeviceIsolationTest.php`

### AC-005 — Par de chaves ed25519 gerado no app e chave privada armazenada em SecureStorage
**Dado** que o app realiza o primeiro login
**Quando** o fluxo de device binding é executado
**Então** `@noble/ed25519` gera o par de chaves, a chave privada é armazenada via `Capacitor.Plugins.Preferences` (SecureStorage) e a chave pública é enviada ao backend — verificado em `tests/unit/device/keygen.spec.ts`

### AC-006 — Tela `/admin/devices` lista dispositivos do tenant (somente admin)
**Dado** que o usuário tem role `admin` e existem dispositivos registrados no tenant
**Quando** navega para `/admin/devices`
**Então** a lista exibe nome do dispositivo, plataforma e data de último acesso; um usuário com role `tecnico` recebe HTTP 403 ao tentar acessar a mesma rota — verificado em `tests/e2e/admin-devices.spec.ts`

### AC-007 — Chave privada não aparece em nenhuma chamada de rede
**Dado** que o app está executando
**Quando** o teste intercepta todas as chamadas HTTP feitas pelo app (`tests/unit/device/key-leak.spec.ts`)
**Então** nenhum header, body ou query parameter contém a chave privada ed25519 — exit 0

## Evidências necessárias para aprovação

- Resultado de `php artisan test tests/Feature/Devices/` no backend com exit 0
- Resultado de `npm run test -- tests/unit/device/` no frontend com exit 0
- Resultado de `npm run test:e2e -- tests/e2e/admin-devices.spec.ts` com exit 0
- Tabela `registered_devices` com `tenant_id` em schema PostgreSQL
- Chave privada armazenada apenas em SecureStorage — nenhuma ocorrência em chamadas de rede

## Riscos

- **`@noble/ed25519` pode ter issue de WASM em dispositivos mais antigos** — mitigação: verificar suporte no alvo mínimo (Android API 29, iOS 15); alternativa: usar `SubtleCrypto` nativo do browser para ECDSA P-256 se ed25519 falhar
- **SecureStorage no Android pode ser limpo em factory reset** — mitigação: aceitar como comportamento esperado; factory reset exige novo login e novo registro de dispositivo
