# E15-S10 — Wipe remoto: flag + endpoint admin + UI + lógica de limpeza no cliente

## Objetivo

Permitir que o administrador do laboratório (Persona 1 — Marcelo) revogue remotamente o acesso de um dispositivo e apague todos os dados locais armazenados nele — cenário de perda de dispositivo, demissão de funcionário ou substituição de aparelho. Quando acionado, o dispositivo na próxima vez que se conectar recebe `403 WIPED` e apaga automaticamente banco SQLite local, cache, tokens e redireciona para tela de bloqueio.

Após esta story, o admin consegue clicar em "Revogar" na tela `/admin/devices` (E15-S08) e ter a certeza de que os dados do Kalibrium no dispositivo do ex-funcionário serão apagados na próxima vez que o app tiver conexão.

## Escopo

### Backend (ampliação do endpoint de devices — Laravel)
- `POST /api/v1/devices/{id}/wipe` — disponível somente para role `admin`; define `wiped_at = now()` no registro `registered_devices`; tenant-scoped (admin só pode wipar dispositivos do seu próprio tenant — ADR-0016); audit-logged (registra `user_id` do admin, `device_id`, timestamp e IP em tabela de auditoria ou log estruturado)
- Middleware de autenticação atualizado: ao validar JWT, verificar se `registered_devices.wiped_at IS NOT NULL` para o `device_id` presente no token → se wipado, retornar `HTTP 403` com body `{ "error": "WIPED" }`
- Teste Pest: admin wipa dispositivo → próxima requisição com token daquele device retorna 403 WIPED

### Frontend (React + Capacitor)
- `WipeService.wipe()` (stub preparado em E15-S07, agora implementado):
  - Apagar banco SQLite local (`LocalDb.destroyAll()`)
  - Apagar SecureStorage (JWT, chaves, device key)
  - Limpar cache do Service Worker (se PWA)
  - Redirecionar para `/bloqueado` (tela estática com mensagem "Dispositivo revogado pelo administrador. Entre em contato com o suporte.")
- Tela `/bloqueado`: estática, sem links de navegação, apenas mensagem e contato de suporte
- UI em `/admin/devices`: botão "Revogar" ao lado de cada dispositivo listado (somente admin); modal de confirmação ("Tem certeza? Os dados locais serão apagados."); após confirmação, chama `POST /api/v1/devices/{id}/wipe`; atualiza lista marcando dispositivo como "Revogado"
- Teste automatizado end-to-end do fluxo completo: admin wipa → app do usuário faz requisição → recebe 403 WIPED → `WipeService.wipe()` é chamado → banco é apagado → tela `/bloqueado` é exibida
- Sentry/observabilidade: evento `device.wiped` registrado no Sentry (sem PII — apenas `device_id` e `tenant_id`) quando wipe é acionado pelo cliente

## Fora de escopo

- Notificação push ao usuário quando dispositivo é wipado (E12/E21)
- Wipe em dispositivos que nunca mais abrirem o app (limitação aceita: JWT expira em 30d, garantindo bloqueio eventual)
- Wipe de múltiplos dispositivos em lote — pós-MVP
- Audit log completo em tela de UI — apenas log estruturado nesta story

## Critérios de aceite

### AC-001 — `POST /api/v1/devices/{id}/wipe` define `wiped_at` no registro
**Dado** que o admin faz `POST /api/v1/devices/{uuid}/wipe` com JWT de admin
**Quando** a requisição é processada
**Então** o backend retorna HTTP 200 e `registered_devices.wiped_at` é preenchido com timestamp atual — verificado via Pest em `tests/Feature/Devices/WipeTest.php`

### AC-002 — Admin não pode wipar dispositivo de outro tenant
**Dado** que o admin do tenant A tenta wipar um `device_id` que pertence ao tenant B
**Quando** faz `POST /api/v1/devices/{id_tenant_b}/wipe`
**Então** o backend retorna HTTP 404 (Global Scope filtra o registro e ele não é encontrado) — verificado via Pest

### AC-003 — Requisição com token de dispositivo wipado retorna 403 WIPED
**Dado** que o dispositivo foi wipado (`wiped_at IS NOT NULL`)
**Quando** o app faz qualquer requisição autenticada com o JWT daquele device
**Então** o backend retorna `HTTP 403` com body `{ "error": "WIPED" }` — verificado via Pest

### AC-004 — Técnico sem role `admin` não pode wipar dispositivos
**Dado** que o usuário tem role `tecnico`
**Quando** tenta `POST /api/v1/devices/{id}/wipe`
**Então** o backend retorna HTTP 403 — verificado via Pest

### AC-005 — `WipeService.wipe()` apaga banco SQLite, SecureStorage e cache
**Dado** que o app recebe `403 WIPED`
**Quando** `WipeService.wipe()` é executado
**Então** `LocalDb.destroyAll()` é chamado, SecureStorage é limpo e o cache do Service Worker é purgado — verificado em `tests/unit/wipe/wipe-service.spec.ts` com spies nos três métodos, exit 0

### AC-006 — App redireciona para `/bloqueado` após wipe
**Dado** que `WipeService.wipe()` foi executado
**Quando** o processo de wipe termina
**Então** o app navega para `/bloqueado` e a tela exibe a mensagem de revogação sem links de navegação para outras rotas — verificado em `tests/e2e/wipe-flow.spec.ts`

### AC-007 — Fluxo completo E2E: admin wipa → app detecta → banco apagado → tela bloqueado
**Dado** que o teste `tests/e2e/wipe-e2e.spec.ts` existe
**Quando** se executa `npm run test:e2e -- tests/e2e/wipe-e2e.spec.ts`
**Então** o teste: (1) autentica como admin, (2) wipa o dispositivo de teste via API, (3) simula requisição do app wipado, (4) confirma 403 WIPED recebido, (5) confirma `WipeService.wipe()` invocado (spy), (6) confirma URL em `/bloqueado` — exit 0

### AC-008 — Evento `device.wiped` registrado no Sentry
**Dado** que o wipe é acionado pelo cliente (frontend recebe 403 WIPED)
**Quando** `WipeService.wipe()` é chamado
**Então** `Sentry.captureEvent({ message: 'device.wiped', extra: { deviceId, tenantId } })` é invocado sem PII (sem nome, e-mail ou dados de calibração) — verificado em `tests/unit/wipe/wipe-service.spec.ts`

### AC-009 — Log estruturado local com rotação persistido na tabela `client_logs`
**Dado** que o app registra eventos (info/warn/error) durante operação offline ou online
**Quando** o total de entradas na tabela `client_logs` do SQLite local excede 10.000 registros
**Então** as entradas mais antigas (FIFO) são removidas para que o total permaneça em no máximo 10.000, e cada entrada persistida contém os campos mínimos `(timestamp TEXT, level TEXT, event TEXT, payload_json TEXT, tenant_id TEXT)` — verificado em `tests/unit/logs/client-log-rotation.spec.ts` com `expect(await LocalDb.count('client_logs')).toBeLessThanOrEqual(10000)`, exit 0

### AC-010 — Ping periódico de health enviado ao backend quando o app está online
**Dado** que o app está online e autenticado
**Quando** transcorrem 5 minutos desde o último ping (ou o app volta a ter conexão)
**Então** o app envia `POST /api/v1/health/client-ping` com body `{ app_version: string, last_sync_at: string (ISO-8601), device_id: string }` — verificado em `tests/unit/health/client-ping.spec.ts` com spy no `setInterval` (intervalo de 300.000 ms), mock do `HttpClient.post()` e `expect(mockPost).toHaveBeenCalledWith('/api/v1/health/client-ping', expect.objectContaining({ app_version: expect.any(String), last_sync_at: expect.any(String), device_id: expect.any(String) }))`, exit 0

## Riscos

- **Dispositivo que nunca mais conectar nunca receberá wipe** — mitigação: aceito; JWT expira em 30 dias garantindo bloqueio eventual; documentar para o admin na UI
- **Wipe pode falhar se `LocalDb.destroyAll()` lançar exceção** — mitigação: `WipeService.wipe()` deve ser idempotente; executar em try/finally garantindo que mesmo com erro parcial o SecureStorage seja limpo e o app vá para `/bloqueado`
- **Audit log sem UI pode dificultar auditoria de compliance (ISO 17025)** — mitigação: log estruturado em Sentry cobre MVP; tela de auditoria fica como débito técnico documentado
