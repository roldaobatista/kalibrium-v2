# E15-S09 — Biometria (Face ID / fingerprint) + WebAuthn fallback desktop

## Objetivo

Substituir a inserção de senha a cada abertura do app por autenticação biométrica (Face ID no iOS, fingerprint no Android) após o primeiro login com credenciais. O JWT long-lived (E15-S07) permanece em SecureStorage; a biometria é a "chave" que desbloqueia o acesso a ele. Em dispositivos desktop sem biometria nativa, o fallback é WebAuthn (passkey via hardware key ou TPM). Em situações de falha biométrica (3 tentativas), o app volta ao login com senha.

Após esta story, o técnico de campo (Persona 5 — Beatriz) consegue abrir o app e iniciar trabalho em menos de 3 segundos, sem digitar senha.

## Escopo

- `@capacitor-community/biometric-auth` integrado para iOS (Face ID + Touch ID) e Android (fingerprint + face unlock)
- Fluxo: app abre → verifica JWT válido em SecureStorage → solicita biometria → aprovado → navega para `/home`
- Fallback por PIN de 6 dígitos quando biometria não está disponível no dispositivo (não tem leitor biométrico ou não está cadastrado nas configurações do dispositivo)
- Fallback WebAuthn para navegador desktop: `navigator.credentials.get()` com `publicKeyCredential` usando chave pública do device binding (E15-S08)
- Após 3 falhas biométricas consecutivas → app mostra tela de login com senha (credenciais completas)
- A chave de criptografia AES-256-GCM do banco SQLite (E15-S06) é re-derivada via Argon2id usando resultado da biometria + device key (E15-S08) — substituindo a derivação provisória de E15-S07
- Configuração no `Info.plist` iOS (já preparada em E15-S04): `NSFaceIDUsageDescription` com texto em português
- Tela de configuração em `/perfil/biometria`: habilitar/desabilitar biometria, com aviso de segurança se desabilitada
- Testes: habilitação, fluxo de sucesso, fluxo de falha (3 tentativas), fallback PIN, fallback WebAuthn (mockado)

## Fora de escopo

- Biometria para autorizar transações específicas (ex: assinar certificado de calibração) — pós-MVP
- Cadastro de biometria no dispositivo (responsabilidade do sistema operacional)
- Wipe quando biometria falha permanentemente — E15-S10 cobre wipe remoto; biometria falha local reverte para senha

## Critérios de aceite

### AC-001 — Biometria solicitada ao abrir o app (após JWT válido)
**Dado** que o usuário já fez login anteriormente e o JWT está em SecureStorage válido
**Quando** o app é aberto (ou volta do background)
**Então** o prompt biométrico nativo aparece automaticamente antes de exibir qualquer dado — verificado em `tests/unit/auth/biometric-flow.spec.ts` com mock do plugin (exit 0)

### AC-002 — Autenticação biométrica bem-sucedida navega para `/home`
**Dado** que o prompt biométrico é exibido
**Quando** o usuário autentica com sucesso (Face ID / fingerprint reconhecido)
**Então** o app navega para `/home` sem exibir tela de senha — verificado em `tests/unit/auth/biometric-flow.spec.ts`

### AC-003 — Após 3 falhas biométricas, exibe tela de login com senha
**Dado** que o prompt biométrico é exibido
**Quando** o usuário falha 3 vezes consecutivas na biometria (mock `BiometricAuth.authenticate()` retorna `BIOMETRIC_FAILURE` 3 vezes)
**Então** o app exibe a tela `/login` com campos de e-mail + senha — verificado em `tests/unit/auth/biometric-fallback.spec.ts` com exit 0

### AC-004 — Fallback PIN funciona quando biometria não disponível
**Dado** que o dispositivo não tem biometria cadastrada (`BiometricAuth.isAvailable()` retorna `false`)
**Quando** o app é aberto com JWT válido
**Então** o app solicita PIN de 6 dígitos (tela dedicada) em vez de biometria; PIN correto navega para `/home` — verificado em `tests/unit/auth/pin-fallback.spec.ts`

### AC-005 — WebAuthn funciona no desktop (mock)
**Dado** que o app está no navegador desktop e `Capacitor.isNativePlatform() === false`
**Quando** o usuário precisa de autenticação local
**Então** `navigator.credentials.get()` é chamado (mockado no teste) e o fluxo WebAuthn processa a resposta corretamente — verificado em `tests/unit/auth/webauthn.spec.ts`

### AC-006 — Biometria pode ser desabilitada pelo usuário em `/perfil/biometria`
**Dado** que o usuário está autenticado e navega para `/perfil/biometria`
**Quando** desativa a biometria e confirma
**Então** nas próximas aberturas o app solicita PIN em vez de biometria; o aviso de segurança é exibido durante a desativação — verificado em `tests/e2e/profile-biometric.spec.ts`

### AC-007 — `NSFaceIDUsageDescription` presente no Info.plist iOS
**Dado** que o projeto Xcode está configurado (E15-S04)
**Quando** se executa `grep "NSFaceIDUsageDescription" ios/App/App/Info.plist`
**Então** o resultado retorna o texto de descrição em português — sem essa chave o app seria rejeitado pela Apple

### AC-008 — Re-derivação da chave SQLite com `tenant_id_A` não decifra banco do `tenant_id_B`
**Dado** dois tenants registrados no mesmo dispositivo (cenário simulado em ambiente de dev com dois perfis de SecureStorage distintos, cada um com seu `tenant_id` e `device_key`)
**Quando** a chave AES-256-GCM do banco SQLite do tenant A é re-derivada via Argon2id + biometria + `device_key` do tenant A
**Então** a tentativa de decifrar o banco SQLite do tenant B com essa chave falha com erro de autenticação (GCM auth tag inválida) — verificado em `tests/unit/auth/biometric-key-isolation.spec.ts` com `expect(decryptResult).rejects.toThrow()`, exit 0

## Riscos

- **Android fragmentação biométrica (varia por fabricante)** — mitigação: `@capacitor-community/biometric-auth` abstrai o `BiometricPrompt` da API 28+ que normaliza fabricantes; testar em pelo menos Samsung, Motorola e Xiaomi antes de release
- **Face ID pode não estar cadastrado no dispositivo do usuário** — mitigação: AC-004 cobre o fallback por PIN que é o caminho para qualquer dispositivo sem biometria
- **WebAuthn no Safari pode ter limitações em versões mais antigas** — mitigação: Safari 16.4+ suporta Passkeys nativamente; documentar requisito mínimo no README
