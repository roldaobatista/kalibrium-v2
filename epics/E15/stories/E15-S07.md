# E15-S07 — Autenticação: JWT long-lived + refresh silencioso + tela de login

## Objetivo

Implementar o fluxo completo de autenticação do app mobile/PWA contra o backend Laravel (E02): login com e-mail + senha, recebimento e armazenamento seguro de JWT long-lived (30 dias), refresh silencioso em background antes da expiração, e tela de login com UX Ionic. O JWT, quando válido, habilita as operações offline — o app opera sem conexão por até 30 dias antes de exigir novo login.

Após esta story, o usuário Kalibrium consegue fazer login pelo app, o JWT é armazenado em SecureStorage criptografado, o app atualiza silenciosamente o token antes de expirar, e o usuário é redirecionado para `/home` com o contexto de tenant e role disponível para as demais stores.

## Escopo

- Tela `/login`: campos e-mail + senha, botão "Entrar", loading state, exibição de erro de credenciais inválidas
- Chamada ao endpoint `POST /api/v1/auth/login` do backend E02, recebendo `access_token` (JWT, 30d) e `refresh_token`
- Armazenamento do JWT em `@capacitor/preferences` (SecureStorage nativo no iOS/Android; localStorage fallback no web com aviso de segurança)
- Derivação da chave AES-256-GCM do banco SQLite local (E15-S06) a partir do JWT + PIN do usuário via Argon2id — substituindo a chave temporária gerada em E15-S06
- Refresh silencioso: job em background (Capacitor Background Task ou Web Worker no desktop) que renova o token 24h antes da expiração via `POST /api/v1/auth/refresh`
- Guarda de rota (`AuthGuard`): rota `/home` e demais rotas protegidas redirecionam para `/login` se não há JWT válido em SecureStorage
- Ao receber `401 Unauthorized` em qualquer chamada de API → redirecionar para `/login` e limpar SecureStorage
- Ao receber `403 WIPED` → chamar `WipeService.wipe()` (preparado para E15-S10, stub nesta story)
- Injeção do `tenant_id` extraído do JWT payload no contexto global do `LocalDb` (E15-S06) — toda query local passa a filtrar automaticamente pelo `tenant_id` do usuário logado
- Testes unitários (Vitest): guarda de rota, extração de `tenant_id` do JWT, lógica de refresh, handling de 401/403
- Teste E2E Playwright: fluxo completo login → home → logout → redirecionamento para /login

## Fora de escopo

- Device binding — E15-S08

- Recuperação de senha (pós-MVP ou endpoint do backend já existente)

## Critérios de aceite

### AC-001 — Login com credenciais válidas redireciona para `/home`
**Dado** que existe um usuário cadastrado no backend (`email: tecnico@lab.com`, `password: Senha@123`)
**Quando** o usuário preenche o formulário e submete em `tests/e2e/auth.spec.ts`
**Então** o Playwright confirma que a URL muda para `/home` e o cabeçalho exibe o nome do usuário — exit 0

### AC-002 — JWT é armazenado em SecureStorage após login
**Dado** que o login foi bem-sucedido
**Quando** se verifica o SecureStorage via `Capacitor.Plugins.Preferences.get({ key: 'jwt' })`
**Então** o valor existe, não está em branco e decodifica como JWT com campos `sub`, `tenant_id`, `roles`, `exp` — verificado em `tests/unit/auth/jwt-storage.spec.ts`

### AC-003 — `tenant_id` extraído do JWT é injetado no `LocalDb`
**Dado** que o login foi bem-sucedido e o JWT contém `tenant_id: "tenant-abc"`
**Quando** se chama `LocalDb.query('clientes', {})` sem passar `tenantId` explicitamente após login
**Então** o helper usa automaticamente o `tenant_id` do contexto de autenticação (`"tenant-abc"`) — sem lançar `MissingTenantIdError` — verificado em `tests/unit/auth/tenant-injection.spec.ts`

### AC-004 — Credenciais inválidas exibem erro e não armazenam JWT
**Dado** que o usuário informa senha errada
**Quando** o formulário é submetido
**Então** a tela exibe mensagem de erro ("Credenciais inválidas") sem redirecionar para `/home`, e `Capacitor.Plugins.Preferences.get({ key: 'jwt' })` retorna null — verificado em `tests/e2e/auth.spec.ts`

### AC-005 — AuthGuard redireciona para `/login` sem JWT
**Dado** que não há JWT em SecureStorage
**Quando** o usuário tenta navegar diretamente para `/home`
**Então** é redirecionado para `/login` — verificado em `tests/unit/auth/auth-guard.spec.ts` com exit 0

### AC-006 — Refresh silencioso ocorre antes da expiração
**Dado** que o JWT tem expiração em 24h (simulado no teste com `exp` em 23h59m a partir de agora)
**Quando** o job de refresh é executado
**Então** um novo JWT é obtido via `POST /api/v1/auth/refresh` (mockado no Vitest) e substituído em SecureStorage — verificado em `tests/unit/auth/refresh.spec.ts` com exit 0

### AC-007 — 401 retornado pela API redireciona para `/login` e limpa SecureStorage
**Dado** que o app faz uma chamada autenticada e o backend responde `401 Unauthorized`
**Quando** o interceptor HTTP processa a resposta
**Então** SecureStorage é limpo (`jwt` removido) e o usuário é redirecionado para `/login` — verificado em `tests/unit/auth/http-interceptor.spec.ts`

### AC-008 — `403 WIPED` chama `WipeService.wipe()` (stub)
**Dado** que o backend responde com `403` e body `{ "error": "WIPED" }`
**Quando** o interceptor HTTP processa a resposta
**Então** `WipeService.wipe()` é chamado (spy no teste confirma invocação) — verificado em `tests/unit/auth/http-interceptor.spec.ts`

## Evidências necessárias para aprovação

- Resultado de `npm run test -- tests/unit/auth/` com exit 0
- Resultado de `npm run test:e2e -- tests/e2e/auth.spec.ts` com exit 0

## Riscos

- **Backend E02 pode não retornar `tenant_id` no payload do JWT** — mitigação: verificar no spike E15-S01 se o JWT do backend inclui `tenant_id`; se não incluir, adicionar claim no backend (débito técnico E02) antes de iniciar esta story
- **SecureStorage iOS pode exigir permissão especial** — mitigação: `@capacitor/preferences` usa Keychain no iOS nativamente sem configuração adicional
- **Argon2id pode ser lento em dispositivos mais antigos** — mitigação: medir tempo de derivação em dispositivo alvo mínimo (Android API 29); se > 2s, reduzir parâmetros com nota de segurança
