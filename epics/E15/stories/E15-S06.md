# E15-S06 — Banco local SQLite + SQLCipher + helper de query com enforcement `tenant_id`

## Objetivo

Criar a camada de persistência local offline do Kalibrium: banco SQLite criptografado com SQLCipher, acessado via `@capacitor-community/sqlite`, com um helper de query que aplica automaticamente o filtro de `tenant_id` em toda consulta — espelhando no frontend o mecanismo de Global Scope do backend Laravel (ADR-0016). Para o navegador desktop (sem Capacitor disponível), o fallback é IndexedDB via Dexie com a mesma interface.

Após esta story, o app consegue criar um banco local criptografado, executar migrations e consultar dados com garantia de isolamento por tenant — mesmo que nenhum dado de negócio ainda exista (esses chegam em E16+). O teste de isolamento multi-tenant (cross-tenant leak) é obrigatório e deve passar em CI.

## Escopo

- Integração com `@capacitor-community/sqlite` versão validada em E15-S01, com SQLCipher habilitado
- Chave de criptografia derivada via Argon2id do PIN/biometria (chave temporária para E15-S06: gerada aleatoriamente e armazenada em SecureStorage do Capacitor até E15-S07 estabelecer o fluxo de derivação real)
- IndexedDB via Dexie como fallback para desktop web (mesma interface TypeScript — abstract class `LocalDb`)
- Schema local inicial com as tabelas mapeadas no spike E15-S01, todas com coluna `tenant_id TEXT NOT NULL`
- Migrations locais versionadas (`src/db/migrations/`) com controle de versão (`user_version` no SQLite)
- Helper `LocalDb` (abstract class + duas implementações: `SqliteDb` e `DexieDb`):
  - Método `query<T>(table, filters, tenantId)` que adiciona automaticamente `WHERE tenant_id = ?`
  - Método `insert<T>(table, data, tenantId)` que injeta `tenant_id` automaticamente
  - Não expõe método de query raw sem `tenant_id` — apenas `queryAllTenants()` disponível exclusivamente para conta `super_admin`
- Testes unitários (Vitest) para o helper:
  - Query sem `tenant_id` deve lançar erro `MissingTenantIdError`
  - Query com `tenant_id_A` não retorna registros com `tenant_id_B`
- Teste de integração cross-tenant (`tests/isolation/cross-tenant-leak.spec.ts`): insere registro com `tenant_id_A`, consulta com `tenant_id_B` → deve retornar array vazio
- Teste de criptografia: abrir arquivo `.sqlite` sem a chave (usando sqlite3 CLI externo) → deve retornar `file is not a database` (banco ilegível)

## Fora de escopo

- Sync engine (fila de upload/download, merge por campo, conflito) — E16
- Dados de negócio populados (clientes, OS, calibrações) — E17+
- Derivação real de chave via biometria — E15-S09 complementa com a derivação final
- IndexedDB para dados críticos de calibração em produção — SQLCipher no Capacitor é obrigatório; Dexie só para desktop/notebook (dados menos sensíveis e não persistidos sem Capacitor)

## Critérios de aceite

### AC-001 — Banco SQLite é criado criptografado no dispositivo
**Dado** que o app está rodando em dispositivo/emulador Android com Capacitor
**Quando** o app inicializa pela primeira vez e executa `LocalDb.open(tenantId, key)`
**Então** o banco é criado em storage do app e o teste de criptografia (`tests/unit/db/encryption.spec.ts`) confirma que o arquivo `.sqlite` não é legível sem a chave (retorna erro `file is not a database` ao abrir com sqlite3 CLI sem key) — exit 0

### AC-002 — Migrations executam sem erros e criam todas as tabelas
**Dado** que o banco é criado do zero
**Quando** `LocalDb.runMigrations()` é chamado
**Então** todas as tabelas do schema local existem e `PRAGMA user_version` retorna o número da última migration — verificado em `tests/unit/db/migrations.spec.ts` com exit 0

### AC-003 — Toda tabela do schema local tem coluna `tenant_id`
**Dado** que as migrations foram executadas
**Quando** se executa `PRAGMA table_info(<tabela>)` para cada tabela definida no schema
**Então** a coluna `tenant_id TEXT NOT NULL` existe em 100% das tabelas — verificado via `tests/unit/db/schema.spec.ts`

### AC-004 — Query sem `tenant_id` lança `MissingTenantIdError`
**Dado** que o helper `LocalDb` está instanciado
**Quando** se chama `LocalDb.query('clientes', {})` sem passar `tenantId`
**Então** a chamada lança `MissingTenantIdError` com mensagem descritiva — verificado em `tests/unit/db/tenant-enforcement.spec.ts` com exit 0

### AC-005 — Query com `tenant_id_A` não retorna registros de `tenant_id_B`
**Dado** que existem registros com `tenant_id = 'tenant-A'` e `tenant_id = 'tenant-B'` no banco
**Quando** se chama `LocalDb.query('clientes', {}, 'tenant-A')`
**Então** o resultado contém apenas registros com `tenant_id = 'tenant-A'` e nenhum de `'tenant-B'` — verificado em `tests/unit/db/tenant-enforcement.spec.ts`

### AC-006 — Teste de isolamento cross-tenant (`cross-tenant-leak.spec.ts`) passa com exit 0
**Dado** que o arquivo `tests/isolation/cross-tenant-leak.spec.ts` existe
**Quando** se executa `npm run test -- tests/isolation/cross-tenant-leak.spec.ts`
**Então** exit 0 — o teste insere dados com `tenant_id_A`, consulta com `tenant_id_B` e confirma array vazio retornado

### AC-007 — Fallback Dexie funciona no navegador desktop
**Dado** que o app está rodando em Chrome desktop (sem Capacitor — `Capacitor.isNativePlatform() === false`)
**Quando** `LocalDb.open(tenantId, key)` é chamado
**Então** a implementação `DexieDb` é usada automaticamente (sem SQLCipher), o banco IndexedDB é criado, e `LocalDb.query()` ainda aplica filtro de `tenant_id` — verificado em `tests/unit/db/dexie-fallback.spec.ts` com exit 0

### AC-008 — Teste de criptografia: banco ilegível sem chave
**Dado** que o arquivo `.sqlite` foi criado pelo app com SQLCipher
**Quando** se tenta abrir o arquivo com `sqlite3 <path> .tables` (sem chave SQLCipher)
**Então** o sqlite3 retorna `Error: file is not a database` — evidência documentada no output do `tests/unit/db/encryption.spec.ts`

### AC-009 — CI executa testes de isolamento em todo PR
**Dado** que existe arquivo `.github/workflows/ci.yml` (do E01/E02/E03)
**Quando** um PR toca qualquer arquivo em `src/db/`
**Então** o job `test:isolation` (que roda `tests/isolation/cross-tenant-leak.spec.ts`) é executado automaticamente — verificado pela presença do job no workflow YAML

## Evidências necessárias para aprovação

- Resultado de `npm run test -- tests/unit/db/` com exit 0 (cobrindo AC-001 a AC-008)
- Resultado de `npm run test -- tests/isolation/cross-tenant-leak.spec.ts` com exit 0
- `src/db/migrations/` com pelo menos 1 arquivo de migration e todas as tabelas com `tenant_id`
- Código do helper `LocalDb` com método `query` que adiciona filtro de `tenant_id`
- `.github/workflows/ci.yml` com job `test:isolation` declarado

## Riscos

- **SQLCipher + Capacitor SQLite com bug em iOS 17+** — mitigação: verificado no spike E15-S01; se AC-001 falhar em iOS, ativar plano B documentado no spike (cipher manual libsodium); não bloquear as demais stories de E15
- **Dexie não suporta criptografia nativa** — mitigação: aceito como limitação do modo desktop-web (documentado); dados mais sensíveis (calibração, cliente completo) serão exigidos somente no Capacitor nativo em produção
- **`user_version` conflitar com migrations de E16** — mitigação: estabelecer convenção de numeração de migrations nesta story e documentar para E16
