# E01-S03 — Pipeline CI (PHPStan + Pest + Pint + Rector + SBOM)

## Objetivo
Configurar o pipeline de CI no GitHub Actions que executa automaticamente em todo push e pull request para `main`: análise estática (PHPStan nível 8), testes (Pest), formatação (Pint), refactoring automático (Rector) e geração de SBOM (CycloneDX). Nenhum código chega ao staging sem passar por todos esses gates.

## Escopo
- Workflow GitHub Actions `.github/workflows/ci.yml` com jobs:
  - `lint`: Laravel Pint (`--test`, exit 1 se diff)
  - `static-analysis`: PHPStan nível 8 via `larastan`
  - `rector`: Rector em modo `--dry-run` (exit 1 se diff — sem auto-commit no CI)
  - `tests`: Pest com coverage mínimo configurado (threshold definido no `phpunit.xml`)
  - `sbom`: CycloneDX via `cyclonedx/cyclonedx-php` gerando `sbom.xml` como artefato do workflow
- Cache de dependências no CI: `vendor/` via `actions/cache` com chave `composer.lock`; `node_modules/` com chave `package-lock.json`
- Matrix de PHP: apenas PHP 8.4 (sem matrix multi-versão no MVP)
- Configuração de secrets necessários no workflow (documentados, não expostos): `APP_KEY` de teste gerado no CI
- `phpunit.xml` com configuração de coverage HTML + Clover (artefato do workflow)
- Badge de CI no `README.md` do repositório

## Fora de escopo
- Deploy (E01-S04)
- Scan de vulnerabilidades CVE (Snyk/Dependabot — configuração separada, pós-MVP)
- Matrix multi-versão de PHP
- Notificações de falha de CI por Slack/e-mail
- Self-hosted runners

## Critérios de aceite
- AC-001: Push para branch de feature com código formatado incorretamente faz o job `lint` falhar (exit 1 verificável nos logs do Actions)
- AC-002: Push para branch de feature com erro de tipo PHPStan nível 8 faz o job `static-analysis` falhar
- AC-003: Push para branch de feature com teste falhando faz o job `tests` falhar
- AC-004: Pull request para `main` com todos os jobs verdes exibe check verde no GitHub (status `success` via GitHub API)
- AC-005: Artefato `sbom.xml` é gerado e disponibilizado na aba "Artifacts" do workflow após execução bem-sucedida

## Arquivos/módulos impactados

- `phpunit.xml`
- `phpstan.neon` (ajuste fino se necessário)
- `rector.php` (ajuste fino se necessário)

## Testes obrigatórios
- Teste de integração do pipeline: criar branch com erro de lint intencional → verificar que CI falha (AC-001)
- Teste de integração do pipeline: criar branch com erro PHPStan intencional → verificar que CI falha (AC-002)
- Verificação manual: artefatos `sbom.xml` e `coverage/` presentes na execução do Actions (AC-005)

## Riscos
- Minutos do GitHub Actions em repositório privado são limitados no plano gratuito — mitigação: usar runners padrão (Linux), jobs paralelos para reduzir tempo de wall clock, cache agressivo de `vendor/`
- `cyclonedx/cyclonedx-php` pode não ter suporte completo ao Composer 2 — mitigação: verificar versão compatível antes do merge; alternativa é `cyclonedx-cli` via download direto no step do workflow
- Rector em `--dry-run` pode gerar falsos positivos em código gerado pelo Laravel — mitigação: configurar `rector.php` para excluir `storage/`, `bootstrap/cache/`, `vendor/`

## Dependências
- E01-S01 completa e mergeada (scaffold com `phpstan.neon`, `pint.json`, `rector.php` presentes)
- Repositório GitHub criado com branch `main` protegida (pré-requisito do épico E01)
- Permissão de escrita em `.github/` no repositório

## Rollback esperado
- `git revert` do merge commit desta story
- Workflows são desativados automaticamente quando o arquivo `.yml` é removido
- Nenhum side effect fora do repositório (CI é stateless)
