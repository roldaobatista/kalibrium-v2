# E01-S04 — Deploy staging automatizado (GitHub Actions → VPS)

## Objetivo
Configurar o pipeline de deploy contínuo que, após CI verde em `main`, publica automaticamente a aplicação no VPS Hostinger KVM 1 (ambiente de staging). O PM consegue validar qualquer slice no navegador em staging sem ação manual do agente.

## Escopo
- Workflow GitHub Actions `.github/workflows/deploy-staging.yml` disparado após `ci.yml` verde em `main` (trigger: `workflow_run` com `conclusion: success`)
- Deploy via SSH + rsync (sem Docker no MVP, conforme ADR-0001): copia arquivos para `/var/www/kalibrium-staging/`, executa `composer install --no-dev`, `php artisan migrate --force`, `php artisan config:cache`, `php artisan route:cache`, `php artisan view:cache`, `php artisan queue:restart`
- Secrets do GitHub: `STAGING_SSH_KEY`, `STAGING_HOST`, `STAGING_USER`, `STAGING_PATH` configurados no repositório
- Script `scripts/deploy.sh` versionado no repositório com os steps de deploy (reutilizável localmente)
- Configuração do Nginx no VPS: arquivo de configuração `infra/nginx/kalibrium-staging.conf` (versionado, aplicado manualmente na primeira vez)
- Configuração do PHP-FPM: pool dedicado para staging (`infra/php-fpm/kalibrium-staging.conf`)
- Configuração do Supervisor para Laravel Horizon: `infra/supervisor/horizon-staging.conf` (Horizon instalado e configurado como parte desta story)
- Configuração do Scheduler: entrada de crontab para `php artisan schedule:run` a cada minuto (documentada em `infra/crontab/staging.txt`)
- Laravel Horizon instalado via `composer require laravel/horizon` e publicado (`php artisan horizon:publish`)
- Laravel Scheduler configurado com ao menos um job de sanidade (log heartbeat a cada minuto)
- Monolog-JSON estruturado: `config/logging.php` com channel `stack` usando formatter `JsonFormatter` do Monolog — logs em `storage/logs/laravel.json`
- Variável de ambiente `APP_ENV=staging` e `APP_DEBUG=false` no servidor

## Fora de escopo
- Deploy para produção (KVM 4 — pós-MVP)
- Docker / containerização
- Rollback automatizado (rollback manual via `git revert` + re-push)
- Notificações de deploy por Slack/e-mail
- Blue-green deployment
- CDN para assets estáticos

## Critérios de aceite
- AC-001: Push para `main` com CI verde dispara `deploy-staging.yml` automaticamente — verificável na aba Actions com job `deploy` em status `success`
- AC-002: Após deploy, `curl -s https://staging.kalibrium.com.br/ | grep -c "Laravel"` retorna `1` (ou rota `/` responde HTTP 200 com conteúdo Laravel)
- AC-003: `php artisan horizon:status` no VPS retorna `running` após deploy
- AC-004: `tail -f storage/logs/laravel.json` no VPS mostra linhas em formato JSON válido (não texto plano)
- AC-005: `php artisan schedule:list` no VPS lista o job de heartbeat e ele executa sem erro em `php artisan schedule:run`

## Arquivos/módulos impactados
- `.github/workflows/deploy-staging.yml`
- `scripts/deploy.sh`
- `infra/nginx/kalibrium-staging.conf`
- `infra/php-fpm/kalibrium-staging.conf`
- `infra/supervisor/horizon-staging.conf`
- `infra/crontab/staging.txt`
- `config/logging.php`
- `composer.json` (adição de `laravel/horizon`)
- `app/Console/Kernel.php` ou `routes/console.php` (job de heartbeat)

## Testes obrigatórios
- Integration test do deploy: verificar que o workflow dispara corretamente após CI verde (AC-001 — verificação via GitHub API ou manual)
- Smoke test pós-deploy: `curl` na URL de staging retorna HTTP 200 (AC-002)
- Smoke test de Horizon: `php artisan horizon:status` retorna `running` (AC-003)
- Smoke test de logs: arquivo `laravel.json` é JSON válido linha a linha (AC-004)
- Smoke test de scheduler: `php artisan schedule:run` sem erro (AC-005)

## Riscos
- VPS KVM 1 pode não ter PHP 8.4 instalado por padrão — mitigação: script `infra/scripts/provision-staging.sh` documentado (executado uma única vez pelo PM) para instalar PHP 8.4, Nginx, Redis 8, PostgreSQL 18 client via apt
- Permissões SSH no GitHub Actions podem ser bloqueadas pelo firewall do Hostinger — mitigação: verificar que porta 22 está aberta para IPs do GitHub Actions antes do merge
- `php artisan migrate --force` em staging pode falhar se banco não estiver provisionado — mitigação: dependência explícita de E01-S02 (banco já configurado); verificar conectividade antes do merge

## Dependências
- E01-S02 completa e mergeada (PostgreSQL + Redis configurados)
- E01-S03 completa e mergeada (pipeline CI ativo — pré-requisito do trigger de deploy)
- VPS Hostinger KVM 1 provisionado manualmente pelo PM (SSH ativo, IP disponível)
- DNS `staging.kalibrium.com.br` apontando para o IP do KVM 1
- Secrets `STAGING_SSH_KEY`, `STAGING_HOST`, `STAGING_USER`, `STAGING_PATH` cadastrados nas GitHub Secrets do repositório

## Rollback esperado
- Push de `git revert` para `main` dispara novo deploy automaticamente restaurando o estado anterior
- Para rollback emergencial no servidor: `cd /var/www/kalibrium-staging && git reset --hard <commit-anterior> && composer install --no-dev && php artisan migrate --force`
- Horizon reinicia automaticamente pelo Supervisor após qualquer `queue:restart`

## Evidência necessária para aprovação
- Screenshot da aba "Actions" mostrando `deploy-staging.yml` com status `success`
- Screenshot do navegador abrindo `https://staging.kalibrium.com.br/` e respondendo (qualquer página Laravel)
- Output de `php artisan horizon:status` no VPS mostrando `running`
- Trecho do arquivo `laravel.json` mostrando pelo menos uma linha de log em JSON válido
